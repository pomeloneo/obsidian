## 中文翻译

2024 年 8 月至 9 月初，三个基础设施 bug 间歇性地降低了 Claude 的响应质量。我们已解决这些问题，现在解释发生了什么。

> [!important]
> 
> **明确声明**：我们**从不**因需求、时间或服务器负载降低模型质量。用户报告的问题**完全由基础设施 bug 引起**。

---

### Claude 的大规模服务方式

我们通过第一方 API、Amazon Bedrock 和 Google Cloud Vertex AI 为数百万用户提供 Claude 服务，部署在多个硬件平台：AWS Trainium、NVIDIA GPU 和 Google TPU。

每个硬件平台有不同的特性和优化需求。尽管如此，我们对模型实现有严格的等效标准——用户无论由哪个平台服务都应获得相同质量的响应。

---

### 事件时间线

`[此处有一张图片：Claude API 的事件时间线——黄色：问题检测，红色：恶化，绿色：修复部署]`

第一个 bug 在 8 月 5 日引入，影响约 0.8% 的 Sonnet 4 请求。更多 bug 在 8 月 25-26 日部署中产生。8 月 29 日的负载均衡变更增加了受影响的流量。

---

### 三个重叠问题

#### 1. 上下文窗口路由错误

8 月 5 日起，部分 Sonnet 4 请求被误路由到为即将推出的 [1M token 上下文窗口](https://docs.claude.com/en/docs/build-with-claude/context-windows#1m-token-context-window)配置的服务器。最严重时（8 月 31 日），16% 的 Sonnet 4 请求受影响。约 30% 的 Claude Code 用户有至少一条消息被路由错误。

路由是"粘性"的——一旦请求被错误服务器处理，后续跟进也可能被同一错误服务器处理。

**解决方案**：修复路由逻辑，9 月 4-18 日完成全部部署。

#### 2. 输出损坏

8 月 25 日，TPU 服务器上的错误配置导致 token 生成错误——偶尔给应该很少出现的 token 分配高概率。例如在英文回复中出现泰文或中文字符，或代码中出现明显语法错误。

**解决方案**：9 月 2 日回滚变更。添加了意外字符输出的检测测试。

#### 3. 近似 top-k XLA:TPU 编译错误

8 月 25 日的代码部署意外触发了 XLA:TPU 编译器中的一个潜在 bug。

---

### XLA 编译器 bug 深入分析

Claude 生成文本时会计算每个可能下一个词的概率，然后使用"top-p 采样"。在 TPU 上，概率计算发生在多个芯片的不同位置。

> [!important]
> 
> **根本原因**：混合精度算术。模型以 bf16（16位浮点）计算概率，但 TPU 编译器可能将某些操作优化为 fp32（32位）。这种精度不匹配导致不同操作在最高概率 token 上不一致，使最高概率 token 有时完全从考虑中消失。

`[此处有一张图片：2024 年 12 月 temperature=0 时 bug 的代码补丁截图]`

8 月的修复移除了 12 月的临时解决方案（因为我们认为已解决了根本原因），这暴露了近似 top-k 操作中更深层的 bug——一种性能优化，在某些批量大小和模型配置下会返回完全错误的结果。

`[此处有一张图片：与 XLA:TPU 团队分享的 bug 复现代码截图]`

`[此处有一张图片：与 XLA:TPU 工程师分享的底层近似 top-k bug 复现器截图]`

最终我们发现精确 top-k 操作不再有之前的性能惩罚，于是从近似切换到精确 top-k，并将一些操作标准化为 fp32 精度。**模型质量不可妥协**。

---

### 为什么检测困难

- 评估未能捕获用户报告的退化

- 隐私实践限制了工程师对用户交互的访问

- 每个 bug 在不同平台上产生不同症状

- 过度依赖有噪声的评估

---

### 我们正在改变的

- **更敏感的评估**：开发能更可靠地区分正常与异常实现的评估

- **在更多地方运行质量评估**：在真实生产系统上持续运行

- **更快的调试工具**：在不牺牲用户隐私的情况下更好地调试社区反馈

用户可以通过 Claude Code 中的 `/bug` 命令或 Claude 应用中的"👎"按钮发送反馈。

## 📝 英文原文 / English Original

Between August and early September, three infrastructure bugs intermittently degraded Claude's response quality. We never reduce model quality due to demand, time of day, or server load.

### How we serve Claude at scale

We serve Claude via first-party API, Amazon Bedrock, and Google Cloud Vertex AI, across AWS Trainium, NVIDIA GPUs, and Google TPUs.

### Timeline

First bug introduced August 5 (0.8% of Sonnet 4 requests). More bugs from August 25-26 deployments. Load balancing change on August 29 increased affected traffic.

### Three overlapping issues

**1. Context window routing error:** Short-context requests misrouted to 1M-token servers. At peak, 16% of Sonnet 4 requests affected. Fixed September 4-18.

**2. Output corruption:** Misconfiguration on TPU servers caused high probability for wrong tokens (Thai/Chinese characters in English, syntax errors in code). Rolled back September 2.

**3. Approximate top-k XLA:TPU miscompilation:** Latent compiler bug triggered by sampling code rewrite. The root cause involved mixed precision arithmetic (bf16 vs fp32). Switched to exact top-k.

### Why detection was difficult

Evaluations didn't capture the degradation. Privacy practices limited access to user interactions. Each bug produced different symptoms on different platforms.

### What we're changing

More sensitive evaluations, continuous quality evaluations on production systems, faster debugging tooling.
