# 店铺详情页分享漏斗埋点分析

分析仓库：`/Users/bytedance/gitlab/share-it-all`  
分析目标：梳理「店铺详情页进入 -> 分享入口曝光 -> 点击分享 -> 分享到站外 / 站内 IM / 分享面板底部工具栏 -> 分享回流进入店铺详情页」的埋点漏斗。  
分析范围：SEA 店铺详情页 `shop-store-page-sea` 右上角 Share 入口，也就是 `web_monorepo/.../shopping-guide/shop-showcase-sea` 和 `ttec_shop_api` / `seller_core` 链路。  
排除范围：本文不是橱窗页分析，不使用 `seller-showcase-page`、`showcase_api`、`showcase_share_reflow`、`show_window_share_outside` 这套橱窗页口径。

## 结论

店铺详情页分享不要复用 PDP 的 `tiktokec_share_product` 主口径。店铺页右上角分享自己的主链路是：

```text
tiktokec_enter_shop
  -> tiktokec_module_show(module_name = 'share')
  -> tiktokec_module_click(module_name = 'share')
  -> POST /api/shop/v1/share
  -> Native share JSB / 通用分享面板
  -> tiktokec_shop_share_success(channel_id = JSB 返回 button)
  -> tiktokec_enter_shop(enter_from = shop_share_outside / chat_im)
     + shop_share_reflow 服务端计数校验
```

推荐漏斗层级：

| 漏斗层级             | 推荐事件 / 指标                                                | 关键过滤 / 字段                                                                                                                    | 说明                                                                                             |
| ---------------- | -------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| 1. 进入店铺详情页       | `tiktokec_enter_shop`                                    | `page_name = 'shop'`；`shop_id = seller_id`；`btm` 按默认 tab 区分                                                                  | 店铺详情页进入主分母。页面 `easyTrack.mount()` 也会上报页面性能/进入类点，但分享漏斗主分母建议优先用业务进入店铺点。                          |
| 2. 分享入口曝光        | `tiktokec_module_show`                                   | `module_name = 'share'`；`page_name = 'shop'`；`shop_id`                                                                       | 右上角 Share icon 曝光。`Toolbar` 只有在 `can_share` 为真时渲染 Share。                                       |
| 3. 点击分享入口        | `tiktokec_module_click`                                  | `module_name = 'share'`；`page_name = 'shop'`；`shop_id`                                                                       | 用户点击店铺页 Share icon。它不是 PDP 的 `button_name = 'product_share'`。                                  |
| 4. 分享数据获取        | HTTP 接口 / 服务端 metric                                     | `POST /api/shop/v1/share`；请求字段只有 `seller_id`；服务端 metric `shop_share(type='shop')`                                            | 客户端点击后才请求分享数据。这个不是用户行为埋点，不建议作为前端漏斗层。                                                           |
| 5. 分享数据失败        | `tiktokec_toast_show`                                    | `action_type = 'share'`；`tips_title = 'Something went wrong. Please try again later.'`                                       | `ShopShare` 失败时展示 toast，可作为异常率。                                                                |
| 6. 分享面板渠道 / 工具点击 | `channel_clicked_start/end`、`action_clicked_start/end`   | 渠道用 `platform`；底部工具用 `action_id`；`panel_source = 'share_panel'`；`unique_id` 关联 start/end                                     | Native 通用分享面板口径。渠道点击和底部工具点击要分开，不等同于店铺分享成功。                                                     |
| 7. 店铺分享 JSB 成功   | `tiktokec_shop_share_success`                            | `channel_id = shareRes.button` 或 `shareRes.data.button`；`page_name = 'shop'`；`shop_id`                                       | 前端只在 JSB 返回 `share_code = 1` 时上报。可作为店铺分享成功/执行成功口径，但不等同于第三方 App 内最终投递成功。                        |
| 8. 分享回流进入店铺      | `tiktokec_enter_shop`；辅助看 `shop_share_reflow` 服务端 metric | `tiktokec_enter_shop.enter_from in ('shop_share_outside', 'chat_im')`；服务端 metric：`type in ('shop_share_outside', 'chat_im')` | 用户级漏斗回流优先用客户端进入店铺事件；`shop_share_reflow` 是服务端计数/对账口径。普通站外是 `shop_share_outside`，IM 是 `chat_im`。 |

不建议用于店铺详情页分享主漏斗的点：

| 不建议点 | 原因 |
| --- | --- |
| `tiktokec_share_product` | PDP 商品分享 / Creator PDP JSB 回调口径；店铺详情页 Share 组件没有上报这个事件。 |
| `tiktokec_share_product_to_chat` | PDP 商品卡 IM 发送口径；店铺分享 IM 卡由 `share_dm_info` 交给 Native，没有确认店铺页会上报该点。 |
| `showcase_share_reflow` | 橱窗页回流，不是店铺详情页。 |
| `show_window_share_outside` | 橱窗页外流参数，不是店铺详情页。 |
| `seller_qrscan` | B 端 / Seller share 回流枚举，不是 C 端店铺详情页右上角 Share 主链路。看 C 端分享回流时要排除。 |

## 店铺详情页和橱窗页边界

这次要分析的是店铺详情页，不是达人橱窗页。代码边界如下：

| 页面 | 前端代码 | 后端 / 回流 | 分享主事件 |
| --- | --- | --- | --- |
| 店铺详情页 | `shop-showcase-sea/src/pages/shop-store-page-sea` | `ttec_shop_api`、`seller_core`、`shop_share_reflow` | `tiktokec_module_click(module_name='share')`、`tiktokec_shop_share_success` |
| 橱窗页 | `seller-showcase-page` / showcase 相关包 | `showcase_api`、`showcase_share_reflow` | showcase 自己的分享/回流口径 |

判断是不是拿错链路，优先看下面几个特征：

```text
店铺详情页：
page_name = 'shop'
shop_id = seller_id
POST /api/shop/v1/share
tiktokec_shop_share_success
shop_share_reflow

橱窗页：
showcase_api
seller-showcase-page
showcase_share_reflow
show_window_share_outside
```

## 分享入口曝光和点击

店铺页 toolbar 代码中，Share icon 的展示条件是首屏数据里的 `can_share`：

```tsx
{can_share && <Share shareTipsMsg={share_tips?.msg} color={itemColor} scene={ShareScene.ICON} />}
```

`share_tips.msg` 只影响 tooltip / 红点，不决定 Share icon 是否展示。

主入口曝光：

```text
event = 'tiktokec_module_show'
module_name = 'share'
page_name = 'shop'
shop_id = seller_id
```

主入口点击：

```text
event = 'tiktokec_module_click'
module_name = 'share'
page_name = 'shop'
shop_id = seller_id
```

代码里还有一个 `ShareScene.ICON_WITH_TEXT` 分支，会走：

```text
tiktokec_button_show / tiktokec_button_click
page_name = 'shop_more_button'
button_name = 'share'
previous_page = 'shop'
```

但当前店铺详情页 toolbar 实际渲染的是 `ShareScene.ICON`，所以看右上角 Share icon 主漏斗时，不要把 `shop_more_button` 分支混作主入口。`shop_more_button` 更适合做兼容/观察口径。

## 点击后获取分享数据

点击 Share 后，前端先请求店铺分享接口：

```text
POST /api/shop/v1/share
request.seller_id = 当前店铺 seller_id
```

后端链路：

```text
shop-store-page-sea Share.handleClick
  -> hertzClient.ShopShare({ seller_id })
  -> ttec_shop_api ShopShare
  -> seller_core GetShareInfo
  -> ShopShareService.GetShopShareInfo
```

接口回包字段：

| 字段 | 用途 |
| --- | --- |
| `share_link` | 普通店铺分享链接。无 `affiliate_share_link` 时作为 JSB `url`。 |
| `affiliate_share_link` | 佣金/联盟分享链接。存在时优先作为 JSB `url`。 |
| `im_card` | 站内 IM 分享卡数据，拼入 JSB `share_dm_info.share_data`。 |
| `share_msg` | 分享文案，作为 `desc`，Android 还会写入 `title`。 |
| `share_poster` | 佣金分享自定义面板数据。只有 `affiliate_share_link` 存在时才用于 `shareCustomPanel`。 |

前端 JSB 参数核心逻辑：

```text
url = affiliate_share_link ? affiliate_share_link : share_link
use_origin_link = !affiliate_share_link
shareJsbMode = 3
type = 'custom'
platform = 'share_native'
enter_from = 'shop'
share_dm_info.share_type = 103
disableAppendParam = 1
```

如果 `ShopShare` 失败，前端不上报分享成功，只会上报失败 toast：

```text
event = 'tiktokec_toast_show'
action_type = 'share'
tips_title = 'Something went wrong. Please try again later.'
```

## 店铺分享成功点

店铺页自己的分享成功事件是：

```text
event = 'tiktokec_shop_share_success'
channel_id = JSB 返回的 button
user_id = 当前 user_id
```

触发条件：

```text
Android: shareRes.share_code === 1
iOS:     shareRes.data.share_code === 1
```

代码只在 `share_code = 1` 时调用 `bizShopShareSuccess(button)`。因此这个点可以作为「店铺分享 JSB 成功 / 客户端分享执行成功」口径，但不要命名成「第三方 App 最终发送成功」，因为本地代码只能确认 JSB 成功回调，不能确认站外 App 内最终投递。

`channel_id` 来自 JSB 返回的 `button`。做渠道拆分前，必须先用真实日志枚举 `channel_id`，再配置站外、IM、Copy、其他工具的映射；不要直接套 PDP 的 `platform` 枚举。

## 如何区分站内 IM 和站外渠道

店铺页不能用 PDP 的 `chat_type` 来区分站内/站外，因为店铺页没有上报 `tiktokec_share_product(chat_type=...)`。这里要分三层看：

| 层级 | 站内 IM | 站外渠道 |
| --- | --- | --- |
| 分享成功源端 | `tiktokec_shop_share_success.channel_id in ('chat_merge', 'chat_more')` | `tiktokec_shop_share_success.channel_id` 为 Native 外部平台枚举 |
| 分享面板点击诊断 | IM 联系人/会话点击不走 `channel_clicked_start` 主口径；可观察 `im_share_head_show`、`tiktokec_im_share_head_show`、`share_head_click_total`，但店铺维度需样本日志确认 | `channel_clicked_start/end.platform` 为外部平台枚举 |
| 回流落地 | `tiktokec_enter_shop.enter_from = 'chat_im'`；服务端对账 `shop_share_reflow(type='chat_im')` | `tiktokec_enter_shop.enter_from = 'shop_share_outside'`；服务端对账 `shop_share_reflow(type='shop_share_outside')` |

`chat_merge` / `chat_more` 的来源是 Native bridge 对 IM 通知的处理：

- `AWEIMShareMessageDidSendNotification` 触发后，JSB 回调 `button = 'chat_merge'`。
- `AWEIMShareMessageDidClickMoreNotification` 触发后，JSB 回调 `button = 'chat_more'`。

如果看板要严格定义“IM 消息已发送”，`chat_merge` 更接近发送通知口径；`chat_more` 是点击 IM More 分支后被 bridge 当成 `share_code=1` 返回，建议作为“站内 IM 分支成功/进入更多联系人链路”统计，是否并入“发送成功”需要结合样本日志或产品定义确认。

外部平台枚举来自 Native `TTKSharePlatform`，常见值包括：

```text
facebook
twitter
instagram
instagram_story
instagram_dm
whatsapp
whatsapp_status
messenger
line
kakaotalk
kakao_story
snapchat_story
snapchat_chats
band
naver_cafe
naver_blog
sms
email
viber
vk
zalo
telegram
reddit
lemon8
tiktok
pinterest
youtube
whatsapp_business
upscrolled
more
```

注意：`messenger`、`sms`、`email`、`instagram_dm`、`snapchat_chats` 这些虽然有“消息/私信”语义，但它们是站外 App / 系统渠道，不是 TikTok 站内 IM。店铺站内 IM 按 `chat_merge` / `chat_more` 和回流 `chat_im` 识别。

## 三分支漏斗

如果看板目标是从「店铺详情页点击分享」开始，向后拆成站外、站内、底部工具栏三条线，建议共同起点统一为：

```text
event = 'tiktokec_module_click'
module_name = 'share'
page_name = 'shop'
shop_id = seller_id
```

共同起点去重建议：

```text
distinct user_id, shop_id, page_session_id/时间窗
```

如果 native 面板事件没有稳定带 `shop_id`，用 `user_id + 时间窗` 从分享点击向后关联；不要假设 `tiktokec_module_click`、`channel_clicked_*`、`tiktokec_shop_share_success` 之间一定有统一的 panel session id。Native 的 `unique_id` 只稳定用于 `channel_clicked_start/end` 或 `action_clicked_start/end` 自身配对。

### 分支 1：分享到站外渠道

站外分享建议拆成三层：渠道按钮点击、店铺分享 JSB 成功、站外回流。

| 漏斗层级        | 推荐事件 / 指标                     | 关键过滤 / 字段                                                                              | 是否主口径      |
| ----------- | ----------------------------- | -------------------------------------------------------------------------------------- | ---------- |
| 0. 点击分享入口   | `tiktokec_module_click`       | `module_name = 'share'`                                                                | 是          |
| 1. 站外渠道点击开始 | `channel_clicked_start`       | `platform` 为站外渠道；`panel_source = 'share_panel'`                                        | 辅助/诊断      |
| 2. 站外渠道点击结束 | `channel_clicked_end`         | 同 `platform`；用 `unique_id` 关联 start                                                    | 辅助/诊断      |
| 3. 店铺分享成功   | `tiktokec_shop_share_success` | `channel_id` 映射到站外渠道                                                                   | 是          |
| 4. 站外回流进入店铺 | `tiktokec_enter_shop`         | `enter_from = 'shop_share_outside'`；可辅看 `shop_share_reflow(type='shop_share_outside')` | 是，用户级回流主口径 |

普通店铺分享的站外回流参数由 `seller_core/rpc/short_link.go` 写入：

```text
source_previous_page = 'share_outside'
enter_method = 'share_outside_link'
enter_from = 'shop_share_outside'
```

因此普通站外分享回流在客户端进入店铺点上应能看到：

```text
tiktokec_enter_shop
enter_from = 'shop_share_outside'
```

同时会命中 `ttec_shop_api` 首屏接口里的服务端计数：

```text
shop_share_reflow
type = 'shop_share_outside'
```

站外分支推荐指标：

| 指标        | 计算口径                                                                                                      |
| --------- | --------------------------------------------------------------------------------------------------------- |
| 站外渠道点击率   | `channel_clicked_start(站外 platform) UV / 店铺分享点击 UV`                                                       |
| 站外分享成功率   | `tiktokec_shop_share_success(站外 channel_id) UV / channel_clicked_start(站外 platform) UV`                   |
| 站外回流率     | `tiktokec_enter_shop(enter_from='shop_share_outside') UV / tiktokec_shop_share_success(站外 channel_id) UV` |
| 站外回流服务端对账 | `shop_share_reflow(type='shop_share_outside') PV`                                                         |

注意：`channel_clicked_start` 是第一排分享渠道按钮点击，不是分享成功；`tiktokec_shop_share_success` 是 JSB 成功回调，不是第三方 App 最终发送成功。

### 分支 2：分享到站内 IM

店铺页分享会把 `im_card` 塞进 JSB：

```text
share_dm_info.share_type = 103
share_dm_info.share_data.content_pb = im_card.data
share_dm_info.share_data.scene = im_card.scene
```

IM 卡的回流参数由 `seller_core/service/shop_share.go` 写入：

```text
source_previous_page = 'message'
enter_method = 'im_card'
enter_from = 'chat_im'
```

因此站内 IM 回流会命中：

```text
tiktokec_enter_shop
enter_from = 'chat_im'
```

同时服务端首屏接口会计数：

```text
shop_share_reflow
type = 'chat_im'
```

店铺页没有确认到 PDP 那种专用 IM 发送事件 `tiktokec_share_product_to_chat`，并且 IM 联系人/会话点击不是 `channel_clicked_start` 这类外部平台点击。IM 线建议这样搭：

| 漏斗层级           | 推荐事件 / 指标                                            | 关键过滤 / 字段                                                        | 是否主口径      |
| -------------- | ---------------------------------------------------- | ---------------------------------------------------------------- | ---------- |
| 0. 点击分享入口      | `tiktokec_module_click`                              | `module_name = 'share'`                                          | 是          |
| 1. IM 联系人/会话曝光 | `im_share_head_show` / `tiktokec_im_share_head_show` | 需样本日志确认是否稳定带 `page_name='shop'` / `shop_id`                      | 辅助/诊断      |
| 2. IM 联系人/会话点击 | `share_head_click_total`                             | `chat_type`、`to_user_id` 等 IM 字段；店铺维度需样本日志确认                     | 辅助/诊断      |
| 3. 店铺 IM 分享成功  | `tiktokec_shop_share_success`                        | `channel_id in ('chat_merge', 'chat_more')`                      | 是          |
| 4. IM 回流进入店铺   | `tiktokec_enter_shop`                                | `enter_from = 'chat_im'`；可辅看 `shop_share_reflow(type='chat_im')` | 是，用户级回流主口径 |

IM 分支推荐指标：

| 指标 | 计算口径 |
| --- | --- |
| IM 分享成功率 | `tiktokec_shop_share_success(channel_id in ('chat_merge','chat_more')) UV / 店铺分享点击 UV` |
| IM 回流率 | `tiktokec_enter_shop(enter_from='chat_im') UV / tiktokec_shop_share_success(channel_id in ('chat_merge','chat_more')) UV` |
| IM 回流服务端对账 | `shop_share_reflow(type='chat_im') PV` |

不要把下面两个 PDP 商品分享点拿来当店铺 IM 主口径：

```text
tiktokec_share_product(platform='chat')
tiktokec_share_product_to_chat(message_type='product_share')
```

本地店铺页代码没有上报它们。

### 分支 3：点击分享面板底部工具栏

底部工具栏是 Native 分享面板第二排 action，不是站外渠道，也不是站内 IM 发送。它的点击点是：

```text
action_clicked_start
action_clicked_end
```

推荐漏斗：

| 漏斗层级        | 推荐事件                          | 关键过滤 / 字段                                                   | 是否主口径                      |
| ----------- | ----------------------------- | ----------------------------------------------------------- | -------------------------- |
| 0. 点击分享入口   | `tiktokec_module_click`       | `module_name = 'share'`                                     | 是                          |
| 1. 底部工具曝光列表 | `show_panel_function`         | `function_show`、`full_channel_list`                         | 辅助                         |
| 2. 底部工具曝光明细 | `exposed_share_panel_actions` | `exposed_action_list`、`function_show`                       | 辅助/诊断                      |
| 3. 底部工具点击开始 | `action_clicked_start`        | `action_id`；`panel_source = 'share_panel'`                  | 是                          |
| 4. 底部工具点击结束 | `action_clicked_end`          | `action_id`；用 `unique_id` 关联 start；iOS 有 `ios_block_status` | 是                          |
| 5. 底部工具执行完成 | `action_clicked_finish`       | `action_id`、`result`                                        | 辅助，native action finish 口径 |

底部工具栏推荐指标：

| 指标 | 计算口径 |
| --- | --- |
| 工具栏曝光率 | `show_panel_function(function_show 非空) UV / 店铺分享点击 UV` |
| 工具栏点击率 | `action_clicked_start UV / 店铺分享点击 UV` |
| 工具栏点击完成率 | `action_clicked_end UV / action_clicked_start UV` |
| 各工具点击占比 | 按 `action_id` 拆分 `action_clicked_start` |

`copy` 需要特别注意：

```text
channel_clicked_start + platform = 'copy'  -> 第一排分享渠道 Copy
action_clicked_start  + action_id = 'copy' -> 第二排底部工具 Copy
```

这两个不要合并。

## 普通分享、佣金分享、Seller Share 分支

店铺分享后端有三类生成分支：

| 分支 | 触发条件 | 链接 / 参数 | 看板建议 |
| --- | --- | --- | --- |
| 普通分享 | 默认 C 端店铺详情页分享，无佣金分享权限或佣金链路不可用 | 返回 `share_link`；前端 `use_origin_link = true`；服务端短链写入 `enter_from = 'shop_share_outside'` | C 端站外分享主口径。 |
| 佣金 / 联盟分享 | `CheckCommissionSharePermission` 通过，并从 `GenPageLinkChainKey` 拿到 `chain_key` 和 `affiliateSchema` | 返回 `affiliate_share_link`、`share_poster`；前端 `use_origin_link = false`，并传 `shareCustomPanel`；originLink 追加 `chain_key` | 可以单独观察，但现有客户端成功点没有显式 `is_commission_share` 字段。 |
| Seller Share | `scene = ShareScene_SellerShare` | 短链 `enter_from = 'seller_qrscan'` | 不是 C 端店铺详情页右上角 Share 主链路。 |

佣金分享的关键点：

1. `seller_core` 在佣金分享分支里把 `chain_key` 追加到店铺 schema。
2. `affiliate_share_link` 的内层 `target_url` 使用：

```text
source_previous_page = 'share_outside'
enter_method = 'share_outside_link'
enter_from = 'linkshare_middlepage'
```

3. `ttec_shop_api` 当前确认的 `shop_share_reflow` 枚举只统计：

```text
shop_share_outside
seller_qrscan
chat_im
```

因此佣金分享从点击侧拆分时要谨慎：`tiktokec_shop_share_success` 本身不带明确的 `is_commission_share` / `affiliate_share_link` 字段。若要做“佣金分享点击成功”漏斗，现有前端埋点不够直接，建议先验证日志里是否有 `chain_key`、`affiliate_traking_key`、`shareCustomPanel` 相关透传；否则需要新增字段。回流侧可以观察 `tiktokec_enter_shop` 是否带 `affiliate_traking_key`，但这代表落地页带 `chain_key`，不等同于源端点击成功。

## 看板推荐结构

### 总览漏斗

| 层级 | 事件 / 指标 | 过滤 |
| --- | --- | --- |
| 进入店铺详情页 | `tiktokec_enter_shop` | `page_name='shop'`, `shop_id` |
| 分享入口曝光 | `tiktokec_module_show` | `module_name='share'` |
| 分享入口点击 | `tiktokec_module_click` | `module_name='share'` |
| 分享数据失败 | `tiktokec_toast_show` | `action_type='share'` |
| 分享 JSB 成功 | `tiktokec_shop_share_success` | `channel_id` 非空 |
| 分享回流 | `tiktokec_enter_shop` | `enter_from in ('shop_share_outside', 'chat_im')` |
| 分享回流服务端对账 | `shop_share_reflow` | `type in ('shop_share_outside', 'chat_im')` |

### 分支看板

| 分支 | 起点 | 过程点 | 转化点 | 回流点 |
| --- | --- | --- | --- | --- |
| 站外 | `tiktokec_module_click(module_name='share')` | `channel_clicked_start(platform=站外渠道)` | `tiktokec_shop_share_success(channel_id=站外渠道)` | `tiktokec_enter_shop(enter_from='shop_share_outside')` |
| 站内 IM | `tiktokec_module_click(module_name='share')` | `im_share_head_show` / `share_head_click_total`，需日志确认店铺维度 | `tiktokec_shop_share_success(channel_id in ('chat_merge','chat_more'))` | `tiktokec_enter_shop(enter_from='chat_im')` |
| 底部工具栏 | `tiktokec_module_click(module_name='share')` | `show_panel_function` / `exposed_share_panel_actions` | `action_clicked_start/end(action_id)` | 通常没有分享回流，不应并入站内/站外回流 |

### SQL / 口径注意

1. `tiktokec_module_show/click`、`tiktokec_shop_share_success` 由店铺页 `LoggingProxy` 上报，初始化全局参数里包含 `page_name='shop'`、`shop_id=seller_id`、`enter_from`、`source_previous_page` 等。
2. Native 通用面板事件是否稳定带 `page_name/shop_id`，需要用样本日志验证。若没有，按 `user_id + 点击分享后的短时间窗` 关联。
3. `channel_clicked_start/end` 和 `action_clicked_start/end` 自身有 `unique_id`，可以用于 start/end 配对；不要拿它去强行关联 `tiktokec_shop_share_success`。
4. `shop_share_reflow` 是服务端 metric，适合做 PV/服务端对账，不适合直接做用户级 UV join。用户级回流漏斗优先使用 `tiktokec_enter_shop` 并按 `enter_from` 过滤。
5. `channel_id` 的站外/IM 分类必须先从真实日志枚举，不能直接猜。

## 事件字典

| 事件 / 指标 | 角色 | 关键字段 | 备注 |
| --- | --- | --- | --- |
| `tiktokec_enter_shop` | 主分母 | `page_name`, `shop_id`, `btm`, `affiliate_traking_key` | `affiliate_traking_key` 只在页面 query 有 `chain_key` 时上报。 |
| `tiktokec_module_show` | 分享入口曝光 | `module_name='share'` | 店铺页右上角 Share icon 曝光。 |
| `tiktokec_module_click` | 分享入口点击 | `module_name='share'` | 店铺页右上角 Share icon 点击。 |
| `tiktokec_button_show/click` | 更多面板兼容分支 | `page_name='shop_more_button'`, `button_name='share'` | 当前 toolbar 主入口不是这个分支。 |
| `tiktokec_toast_show` | 分享数据失败 | `action_type='share'` | `ShopShare` 接口失败。 |
| `channel_clicked_start/end` | 第一排外部分享渠道点击 | `platform`, `unique_id`, `panel_source`, `item_id`, `item_type` | 站外渠道点击诊断；不要用它识别 TikTok 站内 IM。 |
| `im_share_head_show` / `tiktokec_im_share_head_show` | IM 联系人曝光 | `to_user_id`, `chat_type` 等 IM 字段 | 是否稳定带店铺维度需样本日志确认。 |
| `share_head_click_total` | IM 联系人/会话点击 | `chat_type`, `to_user_id`, `rank_index` 等 | TikTok 站内 IM 点击诊断；是否稳定带店铺维度需样本日志确认。 |
| `show_panel_function` | 面板功能曝光 | `function_show`, `full_channel_list`, `filter_channel_list` | 可用于确认当前版本实际渠道和底部工具列表。 |
| `exposed_share_panel_actions` | 底部工具曝光明细 | `exposed_action_list`, `function_show` | 底部工具曝光诊断。 |
| `action_clicked_start/end` | 底部工具点击 | `action_id`, `unique_id`, `panel_source`, `ios_block_status` | 第二排 action，不是站外/IM 分享发送。 |
| `action_clicked_finish` | 底部工具执行完成 | `action_id`, `result` | native action finish 口径，辅助看工具结果。 |
| `tiktokec_shop_share_success` | 店铺分享 JSB 成功 | `channel_id`, `user_id` | 店铺页自己的分享成功点。 |
| `shop_share` | 服务端分享接口 metric | `type='shop'` | `ttec_shop_api.ShopShare` defer 上报，不是前端用户行为。 |
| `shop_share_reflow` | 服务端回流 metric | `type='shop_share_outside'/'chat_im'/'seller_qrscan'` | 店铺分享回流服务端计数；C 端主看 `shop_share_outside` 和 `chat_im`，不要当成用户级 UV 事件直接 join。 |
