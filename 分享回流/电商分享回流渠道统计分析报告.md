# 电商分享回流渠道统计分析报告

分析仓库：`/Users/bytedance/gitlab/share-it-all`  
分析目标：只从「回流回来」角度统计分享带来的回流 UV/PV，覆盖 PDP 商品详情页、店铺详情页、电商直播间、电商短视频。  
口径原则：回流看板优先使用接收端/落地端事件。发送侧事件只用于补充具体渠道归因，不把“点击分享渠道/分享成功”直接当成“回流回来”。

## 总结

回流统计要先分清两个维度：

| 维度 | 含义 | 是否可以直接从落地事件拿到 |
| --- | --- | --- |
| `reflow_channel_group` | 回流大类：站内 IM、站外分享链接、其他/未知 | PDP、店铺、短视频 IM/站外基本可以；直播站外需用样本确认 `livesdk_live_play` 是否落表透传字段 |
| `source_platform` | 具体发送渠道：WhatsApp、Facebook、Messenger、SMS、TikTok IM 等 | 大多数场景不能只靠落地事件。要和发送侧分享事件按 `chain_key`、`share_link_id`、`shareLiveTraceId` 或时间窗 join |

推荐看板主表按下面字段建：

| 字段 | 说明 |
| --- | --- |
| `scene` | `pdp` / `shop` / `live` / `short_video` |
| `landing_event` | 回流落地事件名 |
| `reflow_channel_group` | `im` / `outside` / `other` / `unknown` |
| `source_platform` | 有可靠 join 或落地透传时再填；否则填 `unknown`，不要硬拆 |
| `object_id` | PDP 用 `product_id`，店铺用 `shop_id`，直播用 `room_id`，短视频用 `group_id/item_id` |
| `receiver_user_id` | 回流用户，不是分享发起用户 |
| `share_trace_id` | 可用时填 `chain_key`、`share_link_id`、`from_unique_share_id/shareLiveTraceId` |
| `landing_pv` / `landing_uv` | 回流 PV / UV |

## 场景总表

| 场景 | 回流主事件 | 站内 IM 回流过滤 | 站外回流过滤 | 能否直接拆具体站外 App |
| --- | --- | --- | --- | --- |
| PDP 商品详情页 | `tiktokec_enter_product_detail` | `enter_from_info = 'product_share_im'` 或 `entrance_form = 'product_share_card'` | `enter_from_info = 'product_share_outside'` | 不能。只能直接拆 IM vs 站外；具体 App 需 join `tiktokec_share_product.platform` |
| 店铺详情页 | `tiktokec_enter_shop` | `enter_from = 'chat_im'` | `enter_from = 'shop_share_outside'` | 不能。具体 App 需 join `tiktokec_shop_share_success.channel_id` |
| 电商直播间 | `livesdk_live_play` | `is_ecom > 0` 且 `enter_from_merge = 'chat'`；前置可看 `livesdk_share_chat_click` | 外部链接进房代码透传 `from_share_platform`、`share_from_user_id`，建议先在 `livesdk_live_play` 样本确认字段；确认后按 `from_share_platform` 拆 | 部分可行。`from_share_platform` 落表时可直接拆；否则 join `livesdk_share_v2.share_platform` |
| 电商短视频 | `video_play` | `enter_from = 'chat'` 且 `is_from_dm = 1` | `share_link_id` 非空，或 `ugdp_type` 落表；`ugdp_id` 只作链路 ID，不单独作为分享过滤 | 不能。`ugdp_type` 是打开方式，不是 App；具体 App 需 join `share_video.platform` |

## PDP 商品详情页

PDP 回流主事件：

```sql
event = 'tiktokec_enter_product_detail'
```

### 回流大类

| 回流大类 | 过滤条件 | 说明 |
| --- | --- | --- |
| 站外回流 | `enter_from_info = 'product_share_outside'`，或 `json_extract_scalar(trackParams, '$.enter_from_info') = 'product_share_outside'` | 普通用户从站外分享链接回到 PDP。 |
| 站内 IM 回流 | `enter_from_info = 'product_share_im'`，或 `entrance_form = 'product_share_card'`，或对应 `trackParams` JSON 字段 | IM 商品卡点击回到 PDP。 |

推荐 SQL 过滤：

```sql
-- PDP 站外回流
event = 'tiktokec_enter_product_detail'
and (
  enter_from_info = 'product_share_outside'
  or json_extract_scalar(trackParams, '$.enter_from_info') = 'product_share_outside'
)

-- PDP IM 回流
event = 'tiktokec_enter_product_detail'
and (
  enter_from_info = 'product_share_im'
  or entrance_form = 'product_share_card'
  or json_extract_scalar(trackParams, '$.enter_from_info') = 'product_share_im'
  or json_extract_scalar(trackParams, '$.entrance_form') = 'product_share_card'
)
```

去重建议：

```text
UV: distinct receiver_user_id
对象回流 UV: distinct receiver_user_id, product_id
链路级回流 UV: distinct receiver_user_id, product_id, chain_key
```

### 渠道拆分边界

PDP 落地侧只能稳定拆出：

```text
product_share_outside
product_share_im
```

不要直接把 `product_share_outside` 拆成 WhatsApp/Facebook/Messenger 等具体 App。具体 App 是发送侧的 `tiktokec_share_product.platform`，要做“各站外 App 回流”时必须 join：

```text
发送侧：tiktokec_share_product(chat_type = 'outside', platform)
落地侧：tiktokec_enter_product_detail(enter_from_info = 'product_share_outside')
关联键优先：chain_key；没有 chain_key 再退化 product_id + sender/receiver/time window
```

注意：`shop_share_reflow` 不是 PDP 回流指标，不要用于 PDP 看板。

### PDP 回流承载方式

PDP 回流不只有“全屏 PDP”一种展示形态。PDP 自身埋点里会带 `page_show_type`，常见取值是：

```text
full_screen
half_screen
```

这里的 `half_screen` 就是业务上常说的 PDP 半屏/七分屏承载。PDP 分享回流主口径仍然先看：

```text
tiktokec_enter_product_detail
enter_from_info = product_share_outside / product_share_im
```

代码里还存在“先进入商城，再由商城承接落地页打开 PDP 半屏/七分屏”的路径。`TTKECMallEntryDataUpdateActions` 会处理 `landingPageSchema` 为 `ec/pdp` 的情况：

```text
landingPageSchema endsWith ec/pdp
fullScreen = true  -> full screen PDP
fullScreen = false -> half screen PDP
```

同时这条商城承接路径会按当前商城入口把 PDP 的 `trackParams.enter_from` / `trackParams.enter_from_info` 写成动态来源：

```text
mall
homepage_mall
```

因此取数时要分两层：

| 口径 | 推荐过滤 | 说明 |
| --- | --- | --- |
| PDP 分享直达回流 | `enter_from_info in ('product_share_outside', 'product_share_im')` 或 `entrance_form = 'product_share_card'` | PDP 分享回流主口径。 |
| PDP 分享回流承载形态 | 在主口径上再拆 `page_show_type = 'full_screen' / 'half_screen'` | 用来区分全屏 PDP 和半屏/七分屏 PDP。 |
| 商城承接 PDP 半屏 | `enter_from_info in ('mall', 'homepage_mall')` 且 `page_show_type = 'half_screen'` | 这是商城流量承接 PDP，不要直接并入 PDP 分享回流。只有样本里还能通过 `chain_key`、`source_page_type='product_share'` 或其他分享 trace 证明来自 PDP 分享时，才可单独作为“PDP 分享经商城承接回流”分支。 |

也就是说：可以有“商城首页 + PDP 七分屏”的承载方式，但不要把所有 `homepage_mall/mall + half_screen PDP` 都算作 PDP 分享回流。先用 `product_share_outside/product_share_im` 做主回流，再用 `page_show_type` 拆承载；如果要覆盖商城承接的分享回流，需要先用真实日志确认分享 trace 是否保留下来。

## 店铺详情页

店铺详情页回流主事件：

```sql
event = 'tiktokec_enter_shop'
```

### 回流大类

| 回流大类     | 用户级主口径                                                  | 服务端对账口径                                          | 说明                |
| -------- | ------------------------------------------------------- | ------------------------------------------------ | ----------------- |
| 站外回流     | `tiktokec_enter_shop.enter_from = 'shop_share_outside'` | `shop_share_reflow(type = 'shop_share_outside')` | C 端店铺详情页站外分享链接回流。 |
| 站内 IM 回流 | `tiktokec_enter_shop.enter_from = 'chat_im'`            | `shop_share_reflow(type = 'chat_im')`            | 店铺 IM 卡/消息回流。     |

推荐 SQL 过滤：

```sql
-- 店铺站外回流
event = 'tiktokec_enter_shop'
and enter_from = 'shop_share_outside'

-- 店铺 IM 回流
event = 'tiktokec_enter_shop'
and enter_from = 'chat_im'
```

服务端对账：

```text
metric = shop_share_reflow
type in ('shop_share_outside', 'chat_im')
```

`shop_share_reflow(type='seller_qrscan')` 是 B 端/Seller QR 回流，不属于 C 端店铺详情页右上角 Share 主链路；做 C 端分享回流看板时要排除。

### 渠道拆分边界

店铺落地侧能直接拆：

```text
shop_share_outside
chat_im
```

不能只靠 `tiktokec_enter_shop` 拆 WhatsApp/Facebook/Messenger/SMS。具体发送渠道来自发送侧：

```text
tiktokec_shop_share_success.channel_id
```

已确认站内 IM 发送侧枚举：

```text
chat_merge
chat_more
```

其他如 `messenger`、`sms`、`email`、`instagram_dm`、`snapchat_chats` 虽然有“消息/私信”语义，但它们是站外 App/系统渠道，不是 TikTok 站内 IM。

## 电商直播间

电商直播回流进入主事件：

```sql
event = 'livesdk_live_play'
and cast(is_ecom as int) > 0
```

如果只看明确有商品/挂车直播间，可以收窄为：

```sql
event = 'livesdk_live_play'
and cast(is_ecom as int) = 1
```

### 站内 IM 回流

直播 IM 回流链路中有三层可看：

| 层级 | 事件 | 过滤 | 用途 |
| --- | --- | --- | --- |
| IM 卡曝光 | `livesdk_share_chat_show` | `room_id`、`from_user_id`、`chat_type` | 接收端看到直播分享卡。 |
| IM 卡点击 | `livesdk_share_chat_click` | `room_id`、`from_user_id`、`chat_type` | 接收端点击直播分享卡。 |
| 实际进房 | `livesdk_live_play` | `is_ecom > 0` 且 `enter_from_merge = 'chat'` | 回流进入直播间主口径。 |

辅助曝光事件：

```sql
event = 'livesdk_live_show'
and enter_from_merge = 'chat'
and enter_method = 'live_cover'
```

`livesdk_live_show` 是 IM 直播卡/入口曝光，不是最终进房；最终回流 UV/PV 用 `livesdk_live_play`。

如果 IM 分享消息里带 `shareLiveTraceId`，接收端 `livesdk_live_show` 会带：

```text
from_unique_share_id = shareLiveTraceId
```

这个字段适合把 IM 卡曝光/点击与发送侧分享链路串起来。

### 站外回流

站外直播分享链接进入房间的代码侧特征：

```text
URL/params 同时带 share_app_id 和 share_link_id -> isFromShare = true
contextParams 透传 from_share_platform
contextParams 透传 share_from_user_id
utm_campaign = client_share 时 roomLog.shareSourceType = external_link
```

推荐先在日志样本里确认 `livesdk_live_play` 是否落表以下字段：

```text
from_share_platform
share_from_user_id
from_user_id
enter_from
enter_from_merge
enter_method
```

确认字段落表后，站外回流可按下面口径建：

```sql
event = 'livesdk_live_play'
and cast(is_ecom as int) > 0
and (
  from_share_platform is not null and from_share_platform <> ''
  or share_from_user_id is not null and share_from_user_id <> ''
  or from_user_id is not null and from_user_id <> ''
)
```

如果 `from_share_platform` 有值，可以作为直播站外具体渠道；如果没有值，只能统计“站外分享链接回流”，不能拆到 App。具体渠道要 join 发送侧：

```text
发送侧：livesdk_share_v2.share_platform
落地侧：livesdk_live_play 外部链接回流字段
关联键优先：share_link_id/share_app_id 或 shareLiveTraceId；没有稳定 ID 时只能做时间窗归因，不能作为精确渠道回流
```

## 电商短视频

短视频回流落地第一跳主看播放事件：

```sql
event = 'video_play'
```

电商短视频过滤：

```sql
-- 严格商品视频
is_ecom = 1

-- 更宽的电商视频/电商意图视频，可作为扩展口径
is_ec_video = 1
or cast(is_ecom_intent as int) > 0
or ec_product_cnt > 0
```

### 站内 IM 回流

IM 中打开短视频的落地参数来自聊天内打开 feed 的链路，已确认会写入：

```text
refer = 'chat'
enter_from = 'chat'
is_from_dm = 1
enter_position = 'video_cover'
chat_type
conversation_id
```

推荐 IM 回流过滤：

```sql
event = 'video_play'
and is_ecom = 1
and enter_from = 'chat'
and is_from_dm = 1
```

如果要拆私聊/群聊：

```text
chat_type = 'private' / 'group'
conversation_id
```

### 站外回流

站外短视频分享链接回流会走 deeplink 插入视频逻辑，`video_play` 上会带：

```text
ugdp_id
share_link_id
link_sharer_uid
sharer_info_req_id
ugdp_type
insert_aweme_duration
```

推荐站外回流过滤：

```sql
event = 'video_play'
and is_ecom = 1
and (
  share_link_id is not null and share_link_id <> ''
  or ugdp_type is not null
)
```

注意 `ugdp_type` 不是 WhatsApp/Facebook 这类分享 App，它是打开方式枚举：

```text
1 = AppsFlyer conversion
2 = Universal Link
3 = Scheme
-1 = other
```

因此短视频“站外各 App 回流”不能只用 `video_play.ugdp_type`。要拆具体 App，需要 join 发送侧：

```text
发送侧：share_video(platform, share_link_id)
落地侧：video_play(share_link_id, ugdp_id, ugdp_type)
关联键优先：share_link_id；没有 share_link_id 时不要做精确 App 回流
```

`play_time` 和 `video_play_finish` 也会补充消费侧回流字段：

```text
external_sharer_uid
_oid
```

它们适合看回流后的观看质量，不建议替代 `video_play` 作为“回流进入”的第一跳主口径。

## 从分享到回流的漏斗看板搭建

这个看板不要做成传统的“同一个 user_id 从点击分享到回流”的漏斗。分享侧是发送者，回流侧是接收者，两个 `user_id` 通常不同。建议搭成两张明细事实表，再按 trace 做归因：

```text
share_source_base  -- 发送侧：谁分享了什么、分享到哪个渠道
reflow_landing_base -- 回流侧：谁从分享链路回来了、落到哪个页面
attribution_base   -- 用 chain_key/share_link_id/shareLiveTraceId 等把两侧串起来
dashboard_agg      -- 按 scene/channel/platform/object/date 聚合
```

### 统一字段

发送侧表建议字段：

| 字段 | 说明 |
| --- | --- |
| `scene` | `pdp` / `shop` / `live` / `short_video` |
| `source_event` | 发送侧事件名 |
| `sender_user_id` | 分享发起用户 |
| `object_id` | `product_id` / `shop_id` / `room_id` / `group_id` |
| `channel_group` | `im` / `outside` / `toolbar` |
| `source_platform` | 原始渠道，如 `chat`、`chat_merge`、`whatsapp`、`facebook`、`copy_link` |
| `share_trace_id` | `chain_key` / `share_link_id` / `shareLiveTraceId`，没有就置空 |
| `share_target_cnt` | IM 可用 `chat_cnt` / recipient count；站外通常按 share action PV |
| `source_ts` | 分享时间 |

回流侧表建议字段：

| 字段 | 说明 |
| --- | --- |
| `scene` | 回流落地场景 |
| `landing_event` | 回流落地事件名 |
| `receiver_user_id` | 回流用户 |
| `object_id` | 回流对象 |
| `reflow_channel_group` | `im` / `outside` / `mall_carried` / `unknown` |
| `landing_platform` | 落地侧能直接拿到具体渠道时再填；否则 `unknown` |
| `share_trace_id` | 落地侧保留下来的 trace |
| `page_show_type` | PDP 可拆 `full_screen` / `half_screen` |
| `landing_ts` | 回流时间 |

归因分三档：

| 归因等级 | 规则 | 能否做具体渠道回流 |
| --- | --- | --- |
| P0 精确归因 | `share_trace_id` 能一一 join，例如 `chain_key`、`share_link_id`、`from_unique_share_id/shareLiveTraceId` | 可以 |
| P1 半精确归因 | 落地事件直接带渠道字段，例如直播站外 `from_share_platform` | 可以，但要先确认字段落表 |
| P2 聚合归因 | 没有稳定 trace，只能按 `scene + object_id + channel_group + 时间窗` 聚合比较 | 只能做站内/站外总量，不做具体 App 回流率 |

时间窗建议至少做三档：

```text
D0: landing_ts between source_ts and source_ts + 24h
D1: landing_ts between source_ts and source_ts + 48h
D7: landing_ts between source_ts and source_ts + 7d
```

跨时区看板统一用业务数仓日期，不要把发送侧和回流侧分别按本地日期切开后再相除。

### 看板 A：各渠道分享 -> 回流总览

维度：

```text
date
scene
channel_group
source_platform
attribution_level
```

指标：

| 指标 | 口径 |
| --- | --- |
| 分享 PV | `share_source_base` 行数 |
| 分享 UV | `sender_user_id` 去重 |
| 分享对象数 | `object_id` 去重 |
| 可归因分享 PV | `share_trace_id` 非空，或落地侧有可用渠道字段 |
| 回流 PV | `reflow_landing_base` 行数 |
| 回流 UV | `receiver_user_id` 去重 |
| 回流对象数 | `object_id` 去重 |
| D0/D1/D7 回流率 | `回流 UV / 分享 PV` 或 `回流 UV / share_target_cnt`；IM 优先用 target cnt，站外通常用分享 PV |

注意：站外分享 PV 只是“用户选择/执行站外渠道”的客户端口径，不等于第三方 App 内真实投递或真实曝光。因此站外回流率更准确地叫：

```text
站外分享动作到回流率
```

### 看板 B：PDP 商品详情页分享 -> PDP 回流

推荐分三条分支：

| 分支                    | 发送侧漏斗                                                                                                                                                               | 回流侧漏斗                                                                                                      | 归因方式                                                                            |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| PDP -> 站内 IM -> PDP   | `tiktokec_button_click(button_name='product_share')` -> `tiktokec_share_product(platform='chat')` -> `tiktokec_share_product_to_chat(message_type='product_share')` | `tiktokec_enter_product_detail(enter_from_info='product_share_im' or entrance_form='product_share_card')`  | 优先 `chain_key`；没有则只做 IM 总量                                                      |
| PDP -> 站外 App -> PDP  | `tiktokec_button_click(button_name='product_share')` -> `tiktokec_share_product(chat_type='outside', platform)`                                                     | `tiktokec_enter_product_detail(enter_from_info='product_share_outside')`                                   | 优先 `chain_key`；否则具体 App 不做精确回流                                                  |
| PDP -> 商城承接 -> PDP 半屏 | 发送侧同 PDP 分享                                                                                                                                                         | `tiktokec_enter_product_detail(page_show_type='half_screen', enter_from_info in ('mall','homepage_mall'))` | 只有保留 `chain_key/source_page_type='product_share'` 时才归入 PDP 分享；否则作为商城承接 PDP 流量单列 |

PDP 看板建议指标：

| 指标 | 口径 |
| --- | --- |
| PDP 分享点击 UV | `tiktokec_button_click(button_name='product_share')` |
| IM 发送 UV | `tiktokec_share_product_to_chat(message_type='product_share')` |
| 站外分享 PV/UV | `tiktokec_share_product(chat_type='outside')`，按 `platform` 拆 |
| IM 回流 PDP UV | `tiktokec_enter_product_detail` + IM 回流过滤 |
| 站外回流 PDP UV | `tiktokec_enter_product_detail` + outside 回流过滤 |
| PDP 半屏/七分屏回流占比 | 在 PDP 回流主口径上按 `page_show_type` 拆 |

PDP 关键限制：如果落地侧只有 `product_share_outside`，没有 `chain_key` 或可 join trace，就只能知道“站外回流”，不能知道是 WhatsApp 还是 Facebook 回来的。

### 看板 C：店铺详情页分享 -> 店铺回流

推荐分两条主分支：

| 分支 | 发送侧漏斗 | 回流侧漏斗 | 归因方式 |
| --- | --- | --- | --- |
| 店铺 -> 站内 IM -> 店铺 | `tiktokec_module_click(module_name='share')` -> `tiktokec_shop_share_success(channel_id in ('chat_merge','chat_more'))` | `tiktokec_enter_shop(enter_from='chat_im')`，辅看 `shop_share_reflow(type='chat_im')` | 本地代码未确认稳定 share trace，默认做 IM 总量 |
| 店铺 -> 站外 App -> 店铺 | `tiktokec_module_click(module_name='share')` -> `tiktokec_shop_share_success(channel_id not in ('chat_merge','chat_more'))` | `tiktokec_enter_shop(enter_from='shop_share_outside')`，辅看 `shop_share_reflow(type='shop_share_outside')` | 具体 App 需要发送侧 `channel_id` 与落地 trace/join；没有 trace 时只做站外总量 |

店铺看板建议指标：

| 指标 | 口径 |
| --- | --- |
| 店铺分享点击 UV | `tiktokec_module_click(module_name='share')` |
| 店铺分享成功 UV | `tiktokec_shop_share_success` |
| IM 分享成功 UV | `channel_id in ('chat_merge','chat_more')` |
| 站外分享成功 UV | `channel_id not in ('chat_merge','chat_more')` |
| IM 回流店铺 UV | `tiktokec_enter_shop(enter_from='chat_im')` |
| 站外回流店铺 UV | `tiktokec_enter_shop(enter_from='shop_share_outside')` |
| 服务端回流 PV 对账 | `shop_share_reflow(type in ('chat_im','shop_share_outside'))` |

店铺不要把 `seller_qrscan` 并入 C 端分享回流。它是 Seller/QR 口径。

### 看板 D：电商直播分享 -> 直播回流

推荐分三条分支：

| 分支 | 发送侧漏斗 | 回流侧漏斗 | 归因方式 |
| --- | --- | --- | --- |
| 直播 -> 站内 IM 卡曝光/点击 -> 进房 | `livesdk_user_share_click` / `livesdk_anchor_share_click` -> `livesdk_share_v2(share_platform in ('chat','chat_merge'))`，辅看 `share_video_to_chat` | `livesdk_share_chat_show` -> `livesdk_share_chat_click` -> `livesdk_live_play(is_ecom>0, enter_from_merge='chat')` | `from_unique_share_id/shareLiveTraceId` 可用时精确，否则按 IM 总量 |
| 直播 -> 站外 App -> 进房 | `livesdk_share_v2(to_user_id='out_app_share' or share_platform not in ('chat','chat_merge','bulletin_board'))` | `livesdk_live_play(is_ecom>0)` 且命中 `from_share_platform/share_from_user_id/from_user_id` | `from_share_platform` 落表时可直接拆；否则需要 join |
| 直播 -> 分享卡曝光未进房 | 同 IM 发送侧 | `livesdk_share_chat_show` / `livesdk_share_chat_click`，未命中后续 `livesdk_live_play` | 用于诊断 IM 卡点击到进房损耗 |

直播看板建议指标：

| 指标 | 口径 |
| --- | --- |
| 电商直播间分享点击 UV | 点击分享入口且 `room_id` 在 `livesdk_live_play(is_ecom>0)` 房间集合内 |
| IM 分享 PV/UV | `livesdk_share_v2.share_platform in ('chat','chat_merge')` |
| 站外分享 PV/UV | `livesdk_share_v2.to_user_id='out_app_share'` 或平台不在 IM/公告板枚举 |
| IM 卡曝光 UV | `livesdk_share_chat_show` |
| IM 卡点击 UV | `livesdk_share_chat_click` |
| IM 回流进房 UV | `livesdk_live_play(is_ecom>0, enter_from_merge='chat')` |
| 站外回流进房 UV | `livesdk_live_play(is_ecom>0)` 且命中外部分享落地字段 |

直播站外字段要先用样本确认 `livesdk_live_play` 是否落表 `from_share_platform`。如果没有落表，不能拆站外 App 回流，只能用发送侧 App 分布 + 站外进房总量做对照。

### 看板 E：电商短视频分享 -> 短视频回流

推荐分两条主分支：

| 分支 | 发送侧漏斗 | 回流侧漏斗 | 归因方式 |
| --- | --- | --- | --- |
| 短视频 -> 站内 IM -> 视频播放 | `share_video_to_chat`，或 `share_video(platform='chat_head')` 作为补充 | `video_play(is_ecom=1, enter_from='chat', is_from_dm=1)` | 可按 `chat_type/conversation_id` 做 IM 内部分拆；具体消息级归因需消息/card trace |
| 短视频 -> 站外 App -> 视频播放 | `share_video(platform, share_link_id)` | `video_play(is_ecom=1, share_link_id is not null or ugdp_type is not null)` | 优先 `share_link_id` 精确 join；`ugdp_type` 只能说明打开方式，不是 App |

短视频看板建议指标：

| 指标 | 口径 |
| --- | --- |
| 视频分享 PV/UV | `share_video` / `share_video_to_chat` |
| 站外分享 PV/UV | `share_video.platform` 为站外渠道 |
| IM 分享 PV/UV | `share_video_to_chat` 或 IM 平台补充口径 |
| 站外回流播放 UV | `video_play(is_ecom=1)` 且 `share_link_id` 或 `ugdp_type` 命中 |
| IM 回流播放 UV | `video_play(is_ecom=1, enter_from='chat', is_from_dm=1)` |
| 回流后有效播放 | `play_time`，按 `duration` 阈值做 3s/5s/10s |
| 回流后完播 | `video_play_finish` |
| 回流后电商点击 | 商品锚点曝光/点击事件，按 `group_id + receiver_user_id + 时间窗` 关联 |

短视频如果要做“WhatsApp 回流 UV / Facebook 回流 UV”，必须用 `share_link_id` join `share_video.platform`。只有 `video_play.ugdp_type` 时不能拆 App。

### 推荐最终看板分组

实际 BI 页面建议做 5 个 tab：

| Tab | 内容 | 主要用途 |
| --- | --- | --- |
| `Overview` | 全场景 IM/站外分享与回流总览 | 看总体趋势和场景贡献 |
| `PDP` | PDP 分享点击、IM/站外发送、PDP 回流、全屏/半屏拆分 | 看商品详情页分享回流 |
| `Shop` | 店铺分享成功、IM/站外回流、`shop_share_reflow` 对账 | 看店铺详情页分享回流 |
| `Live` | 直播分享、IM 卡曝光/点击、进房回流 | 看直播 IM 卡和站外进房 |
| `Short Video` | 视频分享、deeplink 播放回流、回流后播放质量 | 看短视频分享带来的电商视频消费 |

## 不要这样统计

1. 不要把发送侧 `share_video`、`tiktokec_share_product`、`tiktokec_shop_share_success`、`livesdk_share_v2` 直接当成回流。它们表示分享动作，不表示有人回来了。
2. 不要把 PDP 的回流算到 `shop_share_reflow`。`shop_share_reflow` 是店铺回流服务端 metric。
3. 不要把 `livesdk_live_show` 当成直播进房。直播回流进入用 `livesdk_live_play`。
4. 不要把 `ugdp_type` 当成短视频具体分享 App。它是 deeplink 打开方式。
5. 不要把 `messenger`、`sms`、`email`、`instagram_dm` 算作 TikTok 站内 IM。它们是站外渠道。
6. 如果没有 `chain_key/share_link_id/shareLiveTraceId` 这类稳定 ID，不建议做“具体站外 App 回流 UV”精确看板，只能做发送侧分布和落地侧站外回流总量。
