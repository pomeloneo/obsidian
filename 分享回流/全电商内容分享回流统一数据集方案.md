# 全电商内容分享回流统一数据集方案

目标：把 PDP、店铺、直播间、电商意图短视频、Creator/Alliance 达人 PDP 的分享链路统一成一套可在 TEA 上搭总漏斗的数据集。

生成时间：2026-05-27

## 结论

第一版“按内容类型分别搭 TEA 漏斗”已经能覆盖单场景分析，但不能回答“所有电商内容分享合在一起的总分享和总回流”。

第二版应新增一张统一归一后的事实表或虚拟事件：

```text
ecom_share_chain_event
```

它不是原始埋点，而是把不同业务事件统一成同一套字段和同一套 stage。TEA 看板只基于这张表搭：

```text
内容消费/进入 -> 分享入口点击 -> 分享动作发起 -> 回流进入
```

其中“回流进入”必须通过 `share_trace_id` 关联发送侧和回流侧，不能用普通 TEA 同用户漏斗直接串原始事件。

## 核心建模

### 1. 统一事件粒度

一行表示一个归一后的链路节点。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `event_time` | timestamp | 归一事件时间 |
| `ds` | string | 分区日期 |
| `stage` | string | `content_enter` / `share_entry_show` / `share_entry_click` / `share_action` / `reflow_enter` |
| `content_type` | string | `pdp` / `shop` / `live` / `short_video` / `creator_pdp` |
| `object_id` | string | 商品、店铺、直播间、视频等业务对象 ID |
| `object_id_type` | string | `product_id` / `shop_id` / `room_id` / `group_id` |
| `channel_type` | string | `im` / `outside` / `copy_or_action` / `unknown` / `all` |
| `share_trace_id` | string | 严格归因用的链路 ID |
| `trace_id_type` | string | `chain_key` / `share_link_id` / `ugdp_id` / `share_live_trace_id` / `fallback` |
| `sender_user_id` | string | 分享发起人 |
| `reflow_user_id` | string | 回流打开人，仅 `reflow_enter` 有意义 |
| `funnel_subject_id` | string | TEA 跨用户漏斗主体，优先用 `share_trace_id` |
| `origin_event_name` | string | 原始埋点名 |
| `origin_event_id` | string | 原始事件唯一 ID，有则保留 |
| `attribution_quality` | string | `strict_trace` / `source_param_only` / `fallback_window` |
| `is_ecom_content` | int | 是否电商内容 |
| `source_params` | map/string | 保留原始关键字段，便于排查 |

### 2. TEA 主体口径

如果 TEA 支持自定义主体，建议主体直接用：

```text
funnel_subject_id = share_trace_id
```

如果 TEA 只能按用户做 funnel，则在这张归一表里把 TEA 用的 `user_id` 写成 `funnel_subject_id`，同时保留真实用户：

```text
user_id_for_tea = funnel_subject_id
sender_user_id = 原始分享人
reflow_user_id = 原始回流人
```

这样 TEA 漏斗计算的是“分享链路 ID 是否完成回流”，而不是“同一个真实用户是否先分享再回流”。这正是跨用户分享回流需要的口径。

### PDP

| stage | 原始事件 | 过滤 | 关键归一字段 |
| --- | --- | --- | --- |
| `content_enter` | `tiktokec_enter_product_detail` | `page_name = product_detail` | `content_type=pdp`, `object_id=product_id` |
| `share_entry_show` | `tiktokec_button_show` | `button_name = product_share` |  |
| `share_entry_click` | `tiktokec_button_click` | `button_name = product_share` | `sender_user_id=user_id` |
| `share_action` | `tiktokec_share_product` | 站外：`chat_type = outside` | `channel_type=outside`, `share_trace_id=chain_key` |
| `share_action` | `tiktokec_share_product_to_chat` | `message_type = product_share` | `channel_type=im`, `share_trace_id=chain_key` |
| `reflow_enter` | `tiktokec_enter_product_detail` | 站外：`enter_from_info = product_share_outside` | `reflow_user_id=user_id`, `share_trace_id=chain_key` |
| `reflow_enter` | `tiktokec_enter_product_detail` | 站内：`enter_from_info = product_share_im` 或 `entrance_form = product_share_card` | `reflow_user_id=user_id`, `share_trace_id=chain_key` |

### 店铺

| stage | 原始事件 | 过滤 | 关键归一字段 |
| --- | --- | --- | --- |
| `content_enter` | `tiktokec_enter_shop` | 店铺详情页进入 | `content_type=shop`, `object_id=shop_id` |
| `share_entry_show` | `tiktokec_module_show` | `module_name = share`, `page_name = shop` |  |
| `share_entry_click` | `tiktokec_module_click` | `module_name = share`, `page_name = shop` | `sender_user_id=user_id` |
| `share_action` | `tiktokec_shop_share_success` | 店铺分享成功/发起 | `channel_type` 由 `channel_id` 或面板点击事件补充 |
| `reflow_enter` | `tiktokec_enter_shop` | 站内：`enter_from = chat_im` | `channel_type=im` |
| `reflow_enter` | `tiktokec_enter_shop` | 站外：`enter_from = shop_share_outside` | `channel_type=outside` |

店铺严格归因优先找 `chain_key` / `affiliate_traking_key` / link 生成侧返回值。如果只有 `enter_from`，只能标记为：

```text
attribution_quality = source_param_only
```

### 直播间

| stage | 原始事件 | 过滤 | 关键归一字段 |
| --- | --- | --- | --- |
| `content_enter` | `livesdk_live_play` | `is_ecom in (1, 2)` | `content_type=live`, `object_id=room_id` |
| `share_entry_click` | `livesdk_user_share_click` |  | `sender_user_id=user_id` |
| `share_action` | `livesdk_share_v2` | 站内：`share_platform in ('chat', 'chat_merge')` | `channel_type=im` |
| `share_action` | `share_video_to_chat` | 直播 IM 发送链路 | `channel_type=im` |
| `share_action` | `livesdk_share_v2` | 站外：`to_user_id = out_app_share` 或非 chat 平台 | `channel_type=outside` |
| `reflow_enter` | `livesdk_live_play` | 站内：`enter_from_merge = chat` | `channel_type=im` |
| `reflow_enter` | `livesdk_live_play` | 站外：`from_share_platform` / `share_from_user_id` / `from_user_id` 任一非空 | `channel_type=outside` |

直播严格归因优先使用 `shareLiveTraceId` / `from_unique_share_id` 一类 trace。没有 trace 时，站内和站外回流只能做来源参数口径。

### 电商意图短视频

| stage | 原始事件 | 过滤 | 关键归一字段 |
| --- | --- | --- | --- |
| `content_enter` | `video_play` | `is_ecom = 1` 或 `is_ec_video = 1` 或 `is_ecom_intent = 1` | `content_type=short_video`, `object_id=group_id` |
| `share_entry_click` | `click_share_button` |  | `sender_user_id=user_id` |
| `share_action` | `share_video_to_chat` | 站内 IM | `channel_type=im` |
| `share_action` | `share_video` | `platform = chat_head` 可补头像一步发送 | `channel_type=im` |
| `share_action` | `share_video` | 站外分享任务执行 | `channel_type=outside`, `share_trace_id=share_link_id/ugdp_id` |
| `reflow_enter` | `video_play` | 站内：`enter_from = chat` 且 `is_from_dm = 1` | `channel_type=im` |
| `reflow_enter` | `video_play` | 站外：`share_link_id is not null` 或 `ugdp_type is not null` | `channel_type=outside` |

短视频严格归因优先级：

```text
share_link_id > sp_root_share_link_id > ugdp_id > fallback
```

### Creator/Alliance 达人 PDP

| stage | 原始事件 | 过滤 | 关键归一字段 |
| --- | --- | --- | --- |
| `content_enter` | `tiktokec_author_add_product_detail_show` | `version = pdp-new` | `content_type=creator_pdp`, `object_id=product_id` |
| `share_entry_click` | `tiktokec_author_add_product_detail_button_click` | `button_for = share` | `entry_position=header` |
| `share_entry_click` | `affiliate_button_click` | `click_for = share`, `module_name = pdp_secondary_sheet_share_button` | `entry_position=bottom` |
| `share_action` | `tiktokec_share_product` | `button is not null`, `button != cancel` | `share_button=button` |
| `reflow_enter` | `tiktokec_enter_product_detail` 或 Creator PDP 展示事件 | `chain_key` 命中 Creator PDP 分享链路 | `content_type=creator_pdp` |

达人 PDP 的站内 / 站外不能直接复用 Native PDP 的 `chat_type`、`platform`、`message_type`。需要先枚举 `tiktokec_share_product.button`，建立映射表：

```text
button -> channel_type
```

达人 PDP 严格回流判断：

```text
回流侧 chain_key = GenCreatorShareLink 生成的 chain_key
且 page_from = PDP_AFFILIATE
```

## 归因规则

### 严格归因

严格归因只认稳定 trace：

```text
share_trace_id 非空
发送侧 share_trace_id = 回流侧 share_trace_id
```

计算指标：

```text
share_trace_cnt = count(distinct share_trace_id where stage = 'share_action')
reflow_trace_cnt = count(distinct share_trace_id where stage = 'reflow_enter')
strict_reflow_rate = reflow_trace_cnt / share_trace_cnt
```

### 来源参数归因

如果回流事件只有 `enter_from_info` / `enter_from` / `from_share_platform`，但没有稳定 trace：

```text
attribution_quality = source_param_only
```

这类可以展示“分享来源回流量”，但不能并入严格回流率分子。

### 窗口兜底归因

不建议作为主口径。只有在没有 trace 且业务需要粗估时，才用：

```text
same object_id
same content_type
same channel_type
reflow_time between share_time and share_time + N days
```

这类必须单独标记：

```text
attribution_quality = fallback_window
```

## SQL 骨架

下面是建模骨架，字段名需要按实际数仓表调整。

```sql
with share_actions as (
  -- PDP 站外
  select
    event_time,
    ds,
    'pdp' as content_type,
    product_id as object_id,
    'product_id' as object_id_type,
    'outside' as channel_type,
    chain_key as share_trace_id,
    'chain_key' as trace_id_type,
    user_id as sender_user_id,
    cast(null as string) as reflow_user_id,
    'share_action' as stage,
    'tiktokec_share_product' as origin_event_name,
    'strict_trace' as attribution_quality
  from app_log
  where event = 'tiktokec_share_product'
    and chat_type = 'outside'
    and chain_key is not null

  union all

  -- PDP 站内
  select
    event_time,
    ds,
    'pdp',
    product_id,
    'product_id',
    'im',
    chain_key,
    'chain_key',
    user_id,
    cast(null as string),
    'share_action',
    'tiktokec_share_product_to_chat',
    'strict_trace'
  from app_log
  where event = 'tiktokec_share_product_to_chat'
    and message_type = 'product_share'
    and chain_key is not null

  union all

  -- 其他内容类型按上方映射继续 union all
  select
    event_time,
    ds,
    content_type,
    object_id,
    object_id_type,
    channel_type,
    share_trace_id,
    trace_id_type,
    sender_user_id,
    cast(null as string),
    'share_action',
    origin_event_name,
    attribution_quality
  from normalized_other_share_actions
),

reflow_events as (
  -- PDP 站外回流
  select
    event_time,
    ds,
    'pdp' as content_type,
    product_id as object_id,
    'product_id' as object_id_type,
    'outside' as channel_type,
    chain_key as share_trace_id,
    'chain_key' as trace_id_type,
    cast(null as string) as sender_user_id,
    user_id as reflow_user_id,
    'reflow_enter' as stage,
    'tiktokec_enter_product_detail' as origin_event_name,
    case when chain_key is not null then 'strict_trace' else 'source_param_only' end as attribution_quality
  from app_log
  where event = 'tiktokec_enter_product_detail'
    and enter_from_info = 'product_share_outside'

  union all

  -- PDP 站内回流
  select
    event_time,
    ds,
    'pdp',
    product_id,
    'product_id',
    'im',
    chain_key,
    'chain_key',
    cast(null as string),
    user_id,
    'reflow_enter',
    'tiktokec_enter_product_detail',
    case when chain_key is not null then 'strict_trace' else 'source_param_only' end
  from app_log
  where event = 'tiktokec_enter_product_detail'
    and (
      enter_from_info = 'product_share_im'
      or entrance_form = 'product_share_card'
    )

  union all

  -- 其他内容类型按上方映射继续 union all
  select
    event_time,
    ds,
    content_type,
    object_id,
    object_id_type,
    channel_type,
    share_trace_id,
    trace_id_type,
    cast(null as string),
    reflow_user_id,
    'reflow_enter',
    origin_event_name,
    attribution_quality
  from normalized_other_reflows
),

trace_map as (
  select
    share_trace_id,
    min_by(sender_user_id, event_time) as sender_user_id,
    min_by(content_type, event_time) as content_type,
    min_by(object_id, event_time) as object_id,
    min_by(channel_type, event_time) as channel_type,
    min(event_time) as first_share_time
  from share_actions
  where share_trace_id is not null
  group by share_trace_id
),

chain_events as (
  select
    event_time,
    ds,
    stage,
    content_type,
    object_id,
    object_id_type,
    channel_type,
    share_trace_id,
    trace_id_type,
    sender_user_id,
    reflow_user_id,
    share_trace_id as funnel_subject_id,
    origin_event_name,
    attribution_quality
  from share_actions

  union all

  select
    r.event_time,
    r.ds,
    r.stage,
    coalesce(m.content_type, r.content_type) as content_type,
    coalesce(m.object_id, r.object_id) as object_id,
    r.object_id_type,
    coalesce(m.channel_type, r.channel_type) as channel_type,
    r.share_trace_id,
    r.trace_id_type,
    m.sender_user_id,
    r.reflow_user_id,
    r.share_trace_id as funnel_subject_id,
    r.origin_event_name,
    case
      when m.share_trace_id is not null then 'strict_trace'
      else r.attribution_quality
    end as attribution_quality
  from reflow_events r
  left join trace_map m
    on r.share_trace_id = m.share_trace_id
)

select *
from chain_events;
```

## TEA 看板搭法

基于 `ecom_share_chain_event` 后，TEA 不再直接选原始事件，而是选同一个归一事件，按 `stage` 过滤：

### 总漏斗

```text
Step 1: event = ecom_share_chain_event, stage = share_entry_click
Step 2: event = ecom_share_chain_event, stage = share_action
Step 3: event = ecom_share_chain_event, stage = reflow_enter, attribution_quality = strict_trace
```

筛选项：

```text
content_type = all / pdp / shop / live / short_video / creator_pdp
channel_type = all / im / outside
attribution_quality = strict_trace / source_param_only
```

### 核心指标

| 指标 | 计算 |
| --- | --- |
| 分享 PV | `count(*) where stage = share_action` |
| 分享 UV | `count(distinct sender_user_id) where stage = share_action` |
| 可归因分享链路数 | `count(distinct share_trace_id) where stage = share_action and attribution_quality = strict_trace` |
| 回流 PV | `count(*) where stage = reflow_enter` |
| 回流 UV | `count(distinct reflow_user_id) where stage = reflow_enter` |
| 严格归因回流链路数 | `count(distinct share_trace_id) where stage = reflow_enter and attribution_quality = strict_trace` |
| 严格回流率 | `严格归因回流链路数 / 可归因分享链路数` |
| trace 覆盖率 | `可归因分享链路数 / 分享 PV 或去重分享动作数` |

## 上线前校验

必须先跑字段覆盖率，不然统一看板会把不稳定字段误当真实回流。

| 场景 | 必查字段 | 校验 |
| --- | --- | --- |
| PDP | `chain_key`, `enter_from_info`, `entrance_form`, `product_id` | 发送侧和回流侧 `chain_key` 覆盖率 |
| 店铺 | `chain_key`, `affiliate_traking_key`, `enter_from`, `shop_id`, `channel_id` | `channel_id` 是否可用，回流是否有 trace |
| 直播间 | `shareLiveTraceId`, `from_unique_share_id`, `room_id`, `enter_from_merge`, `from_share_platform` | trace 是否能从发送侧串到回流侧 |
| 短视频 | `share_link_id`, `sp_root_share_link_id`, `ugdp_id`, `group_id`, `is_from_dm` | 站内/站外回流字段覆盖率 |
| 达人 PDP | `chain_key`, `button`, `page_from`, `product_id` | `button` 枚举映射，`PDP_AFFILIATE` 链路 trace 覆盖 |

## 推荐实施顺序

1. 先建 `share_actions` 和 `reflow_events` 两张明细中间表。
2. 跑各内容类型 trace 覆盖率，确认哪些能进入 `strict_trace`。
3. 建 `ecom_share_chain_event` 归一事件表。
4. 在 TEA 注册这个数据集或虚拟事件，主体用 `funnel_subject_id`。
5. 搭总漏斗：分享入口点击 -> 分享动作 -> 严格归因回流。
6. 再补分内容类型、渠道类型、归因质量的拆分图。

## 风险边界

- 没有 trace 的回流只能展示“分享来源回流量”，不能并入严格归因回流率。
- 站外分享 PV 表示客户端选择/执行站外渠道，不等于第三方 App 内真实发送成功。
- Creator/Alliance 达人 PDP 需要先确认 `button` 枚举，不能直接套 Native PDP 的 `chat_type/platform`。
- TEA 原始漏斗不要直接用真实 `user_id` 串发送和回流，否则会严重低估回流。
