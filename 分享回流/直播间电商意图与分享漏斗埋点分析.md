# 直播间电商意图与分享漏斗埋点分析

分析仓库：`/Users/bytedance/gitlab/TikTok`  
分析目标：通过客户端埋点计算「直播间 -> 带电商意图直播间 -> 点击分享 -> 分享到站外 / 分享到站内私信」漏斗。
口径修订：2026-05-22 根据抓包与代码复查，补充直播间分享到站内私信时也会上报 `share_video_to_chat`。

## 结论

主漏斗建议使用下面这条链路：

| 漏斗层级 | 推荐事件 | 关键过滤/字段 |
| --- | --- | --- |
| 1. 进入直播间 | `livesdk_live_play` | `room_id`, `anchor_id`, `user_id`, `live_session_id`, `live_play_session_id` |
| 2. 带电商意图直播间 | `livesdk_live_play` | `is_ecom > 0`；更严格可只取 `is_ecom = 1` |
| 3. 点击分享入口 | 观众：`livesdk_user_share_click`；主播：`livesdk_anchor_share_click` | 与第 2 层按 `user_id + room_id + live_play_session_id/live_session_id` 串联 |
| 4a. 分享到站内私信 | 抓包可见主口径：`livesdk_share_v2`；IM 发起链路：`share_video_to_chat` | `livesdk_share_v2.share_platform in ('chat', 'chat_merge')`；`share_video_to_chat.platform in ('chat', 'long_press_chat')` |
| 4b. 分享到站外渠道 | `livesdk_share_v2` | `to_user_id = 'out_app_share'` 或 `share_platform not in ('chat', 'chat_merge', 'bulletin_board')` |

不建议把 `livesdk_live_show` 作为「进入直播间」分母。它在代码里大量用于直播卡、天窗、IM 卡等曝光场景，更适合作为入口曝光；真正进房/开播观看成功分母用 `livesdk_live_play` 更稳。

直播间判断「分享到站内私信」时，会同时涉及两类事件：

- `share_video_to_chat`：IM 发送流里的发起发送事件。普通直播间 `AWEShareTargetTypeLive` 会进入这个上报点，抓包能看到是符合代码逻辑的。
- `livesdk_share_v2`：抓包可见的 live 侧分享事件，站内/站外都覆盖，更适合做客户端可观测漏斗。
- `livesdk_share`：代码里仍有上报，但当前抓包未必可见；如果取数表里没有这个事件，不要作为主口径。

如果只判断「是否发起分享到站内私信」，可以看 `share_video_to_chat`；如果要在同一张漏斗里同时比较站内私信和站外渠道，建议用抓包可见的 `livesdk_share_v2`，站内过滤 `share_platform in ('chat', 'chat_merge')`，并用 `share_video_to_chat` 做 IM 链路校验或补充。

## 电商直播识别口径

直播房间模型里电商类型字段是 `ecCommerceTypeValue`：

- `0`：非电商直播间
- `1`：`hasCommerceGoods = true`，直播间有商品，更偏强电商/挂车
- `2`：`commerceInfo.commercePermission = true`，有电商权限/意图，但当前不一定有商品

源码位置：

- `Vendors/IESLiveMTKit/.../RoomModel+ProtobufPhase3.m`：`ecCommerceTypeValue` 由 `hasCommerceGoods` 和 `commerceInfo.commercePermission` 计算。
- `Vendors/IESLiveMTKit/.../GBLAudienceRoomTrackerService.m`：`livesdk_live_play` 上报时写入 `is_ecom = room.ecCommerceTypeValue`，并补充 `ecom_live_rcmd_info`。
- `Modules/TikTokECommerce/.../ECLiveShoppingBagViewModel.m`：`enterRoomEcomType` 使用同样的 0/1/2 逻辑。
- `Modules/TikTokECommerce/.../ECLiveShoppingBagAudienceItemFragment.m` 和 `TTKECLiveShoppingBagAudienceWidget.swift`：把 `is_ecom` 加到 live track context。

计算「带有电商意图的直播间」建议先用：

```text
livesdk_live_play.is_ecom in (1, 2)
```

如果要看「明确挂车/有商品直播间」，再收窄为：

```text
livesdk_live_play.is_ecom = 1
```

注意：部分点击类事件里的 `is_ecom` 可能是布尔值，来源是 `isEcomForAudience`，而 `livesdk_live_play.is_ecom` 是 0/1/2 类型。做漏斗时建议统一归一成 `is_ecom_flag = cast(is_ecom as int) > 0`，或直接以 `livesdk_live_play` 的 `room_id + live_play_session_id` 作为电商房间集合，再关联后续分享事件。

### 站内私信发送

直播间分享到站内私信时，IM 发送流会先上报：

```text
share_video_to_chat
```

关键过滤：

```text
platform in ('chat', 'long_press_chat')
```

普通直播间分享面板创建的是 `AWEShareTargetTypeLive`，代码路径是：

1. `AWELiveMTShareServiceImplV2.m:393-408`：构造 `AWEShareLiveModel`，并设置 `shareContext.targetType = AWEShareTargetTypeLive`。
2. `AWEIMShareAndForwardFlow.m:371`：发送前调用 `AWEIMShareVideoTracker trackShareMsgWithShareImpl`。
3. `AWEIMShareVideoTracker.m:73-89`：`AWEShareTargetTypeLive` 会跳过非直播场景的过滤。
4. `AWEIMShareVideoTracker.m:223-224`：上报 `share_video_to_chat`。

`share_video_to_chat` 关键字段：

- `platform`：通常为 `chat`；长按分享为 `long_press_chat`
- `enter_from`
- `enter_method`
- `panel_source`
- `position`
- `chat_cnt`
- `group_chat_cnt`
- `to_user_id`
- `to_user_id_list`
- `to_group_id_list`
- `share_filter_type_list`
- `content_type`
- `bind_id`

随后 live 侧还会通过统一分享事件上报。抓包里更稳定可见的是：

```text
livesdk_share_v2
```

站内私信过滤：

```text
share_platform in ('chat', 'chat_merge')
```

含义：

- `chat`：单聊
- `chat_merge`：群聊/多选合并发送

两者分工：

- **IM 发起发送口径**：用 `share_video_to_chat`。
- **跨站内/站外统一漏斗口径**：优先用抓包可见的 `livesdk_share_v2`，站内过滤 `share_platform in ('chat', 'chat_merge')`。
- **IM 直播分享补充明细**：看 `share_live` / `share_live_info`。

相关字段：

- `share_platform`
- `to_user_id`
- `share_relation_type`
- `share_follow_status`
- `rank_index`
- `panel_source`
- `is_with_message`
- `is_picture_share`
- `room_id`
- `anchor_id`
- `live_session_id`
- `live_play_session_id`
- `ecom_live_rcmd_info`

源码位置：

- `Modules/TikTokLive/.../AWELiveMTShareServiceImplV2.m`
  - `willSendIMShareMessageWithContext` 调用 `reportShareInfoWithShareType`
- `Modules/TikTokLive/.../AWELiveShareServiceImplHelper.m`
  - 根据 share list 把平台归一成 `chat` 或 `chat_merge`
- `Vendors/IESLiveMTKit/.../IESLiveShareReponseServiceImp.m`
  - 代码里会上报 `livesdk_share`
  - 同时上报 `livesdk_share_info`，用 `bind_id` 关联明文 `to_user_id`
  - 还会上报 `livesdk_share_v2`
- `Modules/TikTokIM/.../AWEIMShareAndForwardFlow.m`
  - `p_finalizeAndSendContainers` 里先调用 `p_trackShareVideoWithShareList`，再通知 live 侧 `willSendIMShareMessageWithContext`
- `Modules/TikTokIM/.../AWEIMShareVideoTracker.m`
  - `AWEShareTargetTypeLive` 会进入 `trackShareMsgWithShareContext`
  - 上报 `share_video_to_chat`

`livesdk_share_v2` 可能出现多条：

- `IESLiveShareReponseServiceImp.m:239-251`：在 `reportShareInfoWithRoomModel` 内部，对非 `chat_merge` 场景补一条 `livesdk_share_v2`。
- `AWELiveShareServiceImplHelper.m:379-430`：对 `shareList` 逐个联系人再上报 `livesdk_share_v2`。

因此单聊也可能抓到两条 `livesdk_share_v2`；多选时还会按联系人展开。做 UV 漏斗时 `COUNT DISTINCT user_id + room_id + live_play_session_id` 不受这个影响；做 PV/发送次数时不能直接 count `livesdk_share_v2` 原始行，需要按一次分享动作或目标对象去重，或者用 `share_video_to_chat` 作为站内私信发起发送口径。

IM 模块还会额外上报：

```text
share_live
share_live_info
```

字段：

- `platform = chat`
- `platform = chat_merge`
- `author_id`
- `room_id`
- `request_id`
- `previous_page`
- `user_type`
- `group_id`
- `bind_id`

源码位置：

- `Modules/TikTokIM/.../AWEIMShareAndForwardLiveImpl.m`
  - 进入转发列表时上报 `share_live(platform=chat_merge)`
  - 选择单个联系人时上报 `share_live_info` + `share_live(platform=chat)`

取数建议：如果只看站内私信发起发送，`share_video_to_chat` 可以作为判断事件；如果主漏斗要同时覆盖站内私信和站外渠道，建议用抓包可见的 `livesdk_share_v2` 做统一口径。`share_live` 用作 IM 链路校验或补充明细。

注意：`AWELiveMTShareServiceImplV2` 的入口方法名是 `willSendIMShareMessageWithContext`，因此这里的 `share_video_to_chat` / `livesdk_share_v2` 更贴近「站内私信发送发起/发送链路上报」。如果看 IM 服务端严格发送成功，需要再补 IM message send response 口径；客户端分享漏斗主链路可以先用这些点。

## 站外分享链路

站外分享抓包可见主口径同样建议看：

```text
livesdk_share_v2
```

站外过滤推荐：

```text
to_user_id = 'out_app_share'
```

或：

```text
share_platform not in ('chat', 'chat_merge', 'bulletin_board')
```

其中 `share_platform` 是实际分享渠道，例如外部 App、copy、QR code、H5 button 等。代码里对站外场景会把：

- `to_user_id` 写为 `out_app_share`
- `share_relation_type` 写为 `out_app_share`
- `share_follow_status` 写为 `out_app_share`

源码位置：

- `Modules/TikTokLive/.../AWELiveMTShareServiceImplV2.m`
  - `willExecuteShareTask` 处理外部分享 task
  - 调用 `reportShareInfoWithShareType`
- `Vendors/IESLiveMTKit/.../IESLiveShareReponseServiceImp.m`
  - `reportShareInfoWithRoomModel` 内部判断没有 `to_user_id` 即站外分享
  - 代码里会上报 `livesdk_share`
  - 对非 `chat_merge` 场景补充 `livesdk_share_v2`，抓包侧优先看这个事件
- `Modules/TikTokLive/.../AWELiveShareServiceImplHelper.m`
  - H5 分享存在直接上报 `livesdk_share` 的链路，若抓包或落表无 `v2`，需单独校验

注意：从源码调用时机看，`livesdk_share_v2` 更准确是「用户已选择分享渠道/客户端开始执行分享」事件，不一定代表站外 App 最终真实分享成功。仓库里没有看到直播间对应的稳定 `livesdk_share_success` 点位；如果漏斗图要表达严格成功率，需要再和服务端 `/room/share/` 成功或平台 SDK 回调口径确认。客户端漏斗建议命名为「分享发送/发起」而不是「站外成功分享」。

## 漏斗取数建议

### 推荐漏斗定义

1. `live_room_uv`
   - 事件：`livesdk_live_play`
   - 去重：`user_id + room_id + live_play_session_id`

2. `ecom_live_room_uv`
   - 事件：`livesdk_live_play`
   - 过滤：`is_ecom in (1, 2)`
   - 去重同上

3. `share_click_uv`
   - 事件：`livesdk_user_share_click`
   - 关联：同一 `user_id + room_id`，并在对应 `livesdk_live_play` 之后
   - 如需要包含主播：union `livesdk_anchor_share_click`

4. `share_to_im_uv`
   - 事件：`livesdk_share_v2`
   - 过滤：`share_platform in ('chat', 'chat_merge')`
   - 关联：同一 `user_id + room_id`，发生在点击分享之后

5. `share_to_external_uv`
   - 事件：`livesdk_share_v2`
   - 过滤：`to_user_id = 'out_app_share'`
   - 备选过滤：`share_platform not in ('chat', 'chat_merge', 'bulletin_board')`
   - 关联：同一 `user_id + room_id`，发生在点击分享之后

### SQL 伪代码

```sql
WITH live_play AS (
  SELECT
    user_id,
    room_id,
    anchor_id,
    live_session_id,
    live_play_session_id,
    event_time AS play_time,
    CAST(is_ecom AS BIGINT) AS is_ecom
  FROM app_log
  WHERE event = 'livesdk_live_play'
),
ecom_live AS (
  SELECT *
  FROM live_play
  WHERE is_ecom IN (1, 2)
),
share_click AS (
  SELECT
    user_id,
    room_id,
    event_time AS click_time
  FROM app_log
  WHERE event = 'livesdk_user_share_click'
),
share_send AS (
  SELECT
    user_id,
    room_id,
    event_time AS send_time,
    share_platform,
    to_user_id
  FROM app_log
  WHERE event = 'livesdk_share_v2'
)
SELECT
  COUNT(DISTINCT CONCAT(e.user_id, ':', e.room_id, ':', e.live_play_session_id)) AS ecom_live_uv,
  COUNT(DISTINCT CONCAT(c.user_id, ':', c.room_id, ':', e.live_play_session_id)) AS share_click_uv,
  COUNT(DISTINCT CASE
    WHEN s.share_platform IN ('chat', 'chat_merge')
    THEN CONCAT(s.user_id, ':', s.room_id, ':', e.live_play_session_id)
  END) AS share_to_im_uv,
  COUNT(DISTINCT CASE
    WHEN s.to_user_id = 'out_app_share'
      OR s.share_platform NOT IN ('chat', 'chat_merge', 'bulletin_board')
    THEN CONCAT(s.user_id, ':', s.room_id, ':', e.live_play_session_id)
  END) AS share_to_external_uv
FROM ecom_live e
LEFT JOIN share_click c
  ON e.user_id = c.user_id
 AND e.room_id = c.room_id
 AND c.click_time >= e.play_time
LEFT JOIN share_send s
  ON c.user_id = s.user_id
 AND c.room_id = s.room_id
 AND s.send_time >= c.click_time;
```

实际落表时建议增加时间窗，例如分享点击和发送限定在同一 `live_play_session_id` 或进房后 30 分钟内，避免用户同一天多次进同一直播间造成串联误差。

## 埋点选择注意点

1. `livesdk_live_show` 不等于直播间进房，它覆盖直播入口/卡片展示，适合做入口曝光，不适合作为本漏斗分母。
2. `livesdk_click_more_button` 在观众分享面板展示后采样上报，适合做辅助观察，不适合作为点击分享主事件。
3. `livesdk_share_head_click` / `share_head_click_total` 是点击 IM 联系人，不是发送成功。直播间站内私信发起发送可以看 `share_video_to_chat`，统一分享漏斗优先看抓包可见的 `livesdk_share_v2`。
4. `share_live` 是 IM 模块自己的直播分享事件，适合校验站内私信发送；跨站内/站外统一漏斗建议用 `livesdk_share_v2`。
5. `share_video_to_chat` 不只用于短视频。普通直播间 `AWEShareTargetTypeLive` 在 IM ShareV2 发送流里也会上报该事件。
6. 电商字段在不同事件可能有 0/1/2 和 boolean 两种形态，漏斗里要先归一化。
