# Triage

让 project issue tracker 上的 issues 经过一个由 triage roles 组成的小型 state machine。

triage 期间发布到 issue tracker 的每条 comment 或 issue **必须**以此免责声明开头：

```
> *This was generated by AI during triage.*
```

## 参考 docs

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 如何编写持久的 agent briefs
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` knowledge base 的工作方式

## Roles

两个 **category** roles：

- `bug` — 某些东西坏了
- `enhancement` — 新功能或改进

五个 **state** roles：

- `needs-triage` — maintainer 需要评估
- `needs-info` — 等待 reporter 提供更多信息
- `ready-for-agent` — 已完全说明，准备交给 AFK agent
- `ready-for-human` — 需要 human implementation
- `wontfix` — 不会被处理

每个经过 triage 的 issue 都应该恰好带有一个 category role 和一个 state role。如果 state roles 冲突，先标记出来并询问 maintainer，再做其他任何事。

这些是 canonical role names — issue tracker 中实际使用的 label strings 可能不同。mapping 应该已经提供给你 - 如果没有，请运行 `/setup-matt-pocock-skills`。

State transitions：unlabeled issue 通常先进入 `needs-triage`；然后从那里移动到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。reporter 回复后，`needs-info` 返回 `needs-triage`。maintainer 可以随时 override — 标记看起来异常的 transitions，并在继续前询问。

## 调用

maintainer 调用 `/triage`，并用自然语言描述他们想要什么。解释请求并行动。示例：

- "Show me anything that needs my attention"
- "Let's look at #42"
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## 展示需要关注的内容

查询 issue tracker，并按最旧优先展示三个 buckets：

1. **Unlabeled** — 从未 triaged。
2. **`needs-triage`** — evaluation 进行中。
3. **`needs-info` with reporter activity since the last triage notes** — 需要重新评估。

展示 counts 和每个 issue 的单行摘要。让 maintainer 选择。

## Triage 特定 issue

1. **收集 context。** 阅读完整 issue（body、comments、labels、reporter、dates）。解析任何先前的 triage notes，避免重新询问已解决的问题。使用项目的 domain glossary 探索 codebase，并遵守该区域的 ADRs。读取 `.out-of-scope/*.md`，并提出任何与此 issue 相似的过往拒绝记录。

2. **推荐。** 告诉 maintainer 你推荐的 category 和 state，并给出理由，再附上与此 issue 相关的简短 codebase summary。等待指示。

3. **复现（仅 bugs）。** 在任何 grilling 之前，尝试 reproduction：阅读 reporter 的 steps，追踪相关代码，运行 tests 或 commands。报告发生了什么 — 带 code path 的成功 repro、失败 repro，或细节不足（强烈的 `needs-info` 信号）。确认的 repro 会让 agent brief 有力得多。

4. **Grill（如需要）。** 如果 issue 需要充实，运行一次 `/grill-with-docs` session。

5. **应用 outcome：**
   - `ready-for-agent` — 发布 agent brief comment（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — 使用与 agent brief 相同的结构，但说明为什么它无法委派（judgment calls、external access、design decisions、manual testing）。
   - `needs-info` — 发布 triage notes（见下面的 template）。
   - `wontfix`（bug）— 礼貌解释，然后 close。
   - `wontfix`（enhancement）— 写入 `.out-of-scope/`，在 comment 中链接它，然后 close（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — 应用该 role。如果已有部分进展，可以选择添加 comment。

## 快速 state override

如果 maintainer 说 "move #42 to ready-for-agent"，相信他们并直接应用该 role。确认你将要做什么（role changes、comment、close），然后行动。跳过 grilling。如果没有 grilling session 就移动到 `ready-for-agent`，询问他们是否想写 agent brief。

## Needs-info template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

将 grilling 期间已解决的所有内容记录在 "established so far" 下，避免工作丢失。Questions 必须具体且 actionable，不能是 "please provide more info"。

## 恢复之前的 session

如果 issue 上存在先前的 triage notes，读取它们，检查 reporter 是否回答了任何未解决问题，并在继续前展示更新后的情况。不要重新询问已解决的问题。
