## 中文翻译

[Model Context Protocol (MCP)](https://modelcontextprotocol.io/docs/getting-started/intro) 可以赋予 LLM 智能体数百个工具来解决实际任务。但如何让这些工具发挥最大效能？

本文介绍我们在多种智能体 AI 系统中提升性能的最有效技术。

---

### 什么是工具？

在传统软件中，我们建立的是**确定性系统之间的契约**——`getWeather("NYC")` 每次都以完全相同的方式获取纽约天气。

工具是一种新型软件，反映的是确定性系统与**非确定性智能体**之间的契约。当用户问"今天需要带伞吗？"，智能体可能调用天气工具、从通用知识回答、甚至先询问位置。

> [!important]
> 
> 我们的目标是扩大智能体能有效使用工具解决各种任务的范围。幸运的是，对智能体最"人体工学"的工具对人类也出奇地直观。

---

### 如何编写工具

#### 构建原型

先快速搭建工具原型。如果使用 [Claude Code](https://www.anthropic.com/claude-code) 编写工具，提供工具依赖的软件库文档会有帮助。将工具包装在[本地 MCP 服务器](https://modelcontextprotocol.io/docs/develop/connect-local-servers)或[桌面扩展](https://www.anthropic.com/engineering/desktop-extensions)（DXT）中，可以在 Claude Code 或 Claude Desktop 中连接和测试。

#### 运行评估

通过运行评估来衡量 Claude 使用工具的效果。

`[此处有一张图片：展示工程师如何使用 Claude Code 评估智能体工具效能的示意图]`

**生成评估任务**——从真实用例中生成提示和响应对。强任务示例：

- "与 Jane 安排下周会议讨论 Acme Corp 项目，附上上次规划会议笔记并预订会议室"

- "客户 ID 9182 报告被重复收费三次，查找所有相关日志条目并判断是否有其他客户受影响"

弱任务示例：

- "与 jane@acme.corp 安排下周会议"

- "搜索 `purchase_complete` 的支付日志"

**运行评估**——使用直接 LLM API 调用以编程方式运行。使用简单的智能体循环（`while` 循环交替 LLM API 和工具调用）。

**分析结果**——观察智能体在哪里困惑或卡住。阅读推理和反馈。分析工具调用指标：大量冗余调用可能建议需要调整分页参数；大量参数错误可能建议工具需要更清晰的描述。

`[此处有一张图片：Slack MCP 服务器人工编写 vs Claude 优化的测试集准确率对比图]`

`[此处有一张图片：Asana MCP 服务器人工编写 vs Claude 优化的测试集准确率对比图]`

#### 与智能体协作

你甚至可以让智能体分析结果并为你改进工具。将评估智能体的转录粘贴到 Claude Code 中。Claude 擅长分析转录并同时重构大量工具。

### 编写高效工具的原则

#### 选择正确的工具

更多工具不一定带来更好的结果。常见错误是工具仅仅包装现有 API 端点。

> [!important]
> 
> **智能体有不同于传统软件的"可供性"**。LLM 智能体有有限的"上下文"，而计算机内存便宜且充足。好的工具应该匹配智能体的工作方式——就像在通讯录中按字母查找联系人，而不是从头到尾逐页阅读。

工具可以合并功能，在底层处理多个离散操作：

- 不要分别实现 `list_users`、`list_events`、`create_event`，考虑实现 `schedule_event` 来查找可用时间并安排活动

- 不要实现 `read_logs`，考虑实现 `search_logs` 只返回相关日志行

- 不要分别实现 `get_customer_by_id`、`list_transactions`、`list_notes`，实现 `get_customer_context` 一次性编译所有信息

#### 为工具命名空间

当智能体访问数十个 MCP 服务器和数百个工具时，命名空间（按通用前缀分组相关工具）可以帮助划分边界。例如按服务（`asana_search`、`jira_search`）和资源（`asana_projects_search`、`asana_users_search`）命名。

#### 返回有意义的上下文

工具实现应只返回**高信号信息**。优先考虑上下文相关性，避免低级技术标识符（如 `uuid`、`256px_image_url`、`mime_type`）。使用 `name`、`image_url`、`file_type` 等更直接指导智能体下游行动的字段。

可以暴露 `response_format` 枚举参数，让智能体控制工具返回 `"concise"` 还是 `"detailed"` 响应。

#### 为工具描述进行提示工程

这是改进工具最有效的方法之一。编写工具描述时，想想你会如何向团队新人描述你的工具——将你隐含带有的专业上下文（特殊查询格式、术语定义、资源间关系）明确化。

Claude Sonnet 3.5 在 [SWE-bench Verified](https://www.anthropic.com/engineering/swe-bench-sonnet) 上取得了最先进的性能，这得益于对工具描述的精确改进。

---

### 展望

要为智能体构建有效的工具，我们需要将软件开发实践从可预测的确定性模式重新定向到非确定性模式。

有效的工具被有意且清晰地定义、审慎使用智能体上下文、可以在多样化工作流中组合、并使智能体能直觉地解决实际任务。

## 📝 英文原文 / English Original

[The Model Context Protocol (MCP)](https://modelcontextprotocol.io/docs/getting-started/intro) can empower LLM agents with potentially hundreds of tools to solve real-world tasks. But how do we make those tools maximally effective?

### What is a tool?

Tools are a new kind of software reflecting a contract between deterministic systems and non-deterministic agents. Our goal is to increase the surface area over which agents can be effective.

### How to write tools

**Building a prototype:** Start by standing up a quick prototype. Wrap tools in a local MCP server or Desktop Extension for testing.

**Running an evaluation:** Generate evaluation tasks grounded in real-world uses. Run programmatically with direct LLM API calls.

**Collaborating with agents:** Let agents analyze results and improve tools. Concatenate transcripts from evaluation agents and paste into Claude Code.

### Principles for writing effective tools

**Choosing the right tools:** More tools don't always lead to better outcomes. Build a few thoughtful tools targeting specific high-impact workflows. Tools can consolidate functionality, handling multiple discrete operations under the hood.

**Namespacing tools:** Group related tools under common prefixes (e.g., `asana_search`, `jira_search`).

**Returning meaningful context:** Prioritize high-signal information. Avoid low-level technical identifiers. Expose a `response_format` parameter for concise vs. detailed responses.
