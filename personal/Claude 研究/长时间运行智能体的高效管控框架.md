## 中文翻译

随着 AI 智能体能力的增强，开发者越来越多地要求它们承担需要数小时甚至数天工作的复杂任务。然而，让智能体在多个上下文窗口间持续推进仍然是一个未解决的问题。

**核心挑战**：长时间运行的智能体必须在离散会话中工作，每个新会话开始时没有之前的记忆。想象一个软件项目由轮班工程师负责，每位新工程师到岗时对上一班发生的事情毫无记忆。

我们开发了一个双重方案：一个**初始化智能体**在首次运行时设置环境，一个**编码智能体**在每次会话中增量推进，同时为下一次会话留下清晰的线索。代码示例见 [quickstart](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding)。

---

### 长时间运行智能体的问题

Claude Agent SDK 具有上下文管理能力（如压缩），理论上可以让智能体无限期地工作。但仅靠压缩是不够的。

> [!important]
> 
> **两种典型失败模式**：
> 
> 1. **一次做太多**：智能体试图一次完成所有功能，上下文用尽时功能半完成且未文档化，下一个会话不得不猜测发生了什么
> 
> 1. **过早宣布完成**：看到已有一些进展后就声称任务完成

**解决思路**：

1. 设置初始环境，为所有功能奠定基础，引导智能体逐步、逐功能地工作

1. 提示每个智能体增量推进，同时在会话结束时让环境保持干净状态

---

### 环境管理

#### 功能列表

为解决一次完成或过早完成的问题，我们提示初始化智能体编写一份详尽的功能需求文件。在 [claude.ai](http://claude.ai) 克隆示例中，这意味着超过 **200 项功能**，如"用户可以打开新对话、输入查询、按回车并看到 AI 回复"。

```json
{
    "category": "functional",
    "description": "New chat button creates a fresh conversation",
    "steps": [
      "Navigate to main interface",
      "Click the 'New Chat' button",
      "Verify a new conversation is created",
      "Check that chat area shows welcome state",
      "Verify conversation appears in sidebar"
    ],
    "passes": false
}
```

> [!important]
> 
> 我们使用 JSON 而非 Markdown 作为功能列表格式，因为模型不太可能不当修改或覆盖 JSON 文件。并使用强措辞指令如"删除或编辑测试是不可接受的"。

#### 增量推进

编码智能体被要求**一次只处理一个功能**。这种增量方法对解决智能体一次做太多的倾向至关重要。

模型需要提交进度到 git 并在进度文件中写入摘要，这使模型可以使用 git 回滚错误的代码更改并恢复代码库的工作状态。

#### 测试

Claude 倾向于在没有适当测试的情况下将功能标记为完成。在构建 Web 应用的案例中，Claude 在被明确提示使用浏览器自动化工具后才能很好地端到端验证功能。

### 快速上手

每个编码智能体被提示执行以下步骤来了解情况：

1. 运行 `pwd` 查看工作目录

1. 阅读 git 日志和进度文件了解最近的工作

1. 阅读功能列表文件，选择最高优先级的未完成功能

典型会话开始时的助手消息流程：

```jsx
[Assistant] 让我先了解项目的当前状态。
[Tool Use] <bash - pwd>
[Tool Use] <read - claude-progress.txt>
[Tool Use] <read - feature_list.json>
[Assistant] 让我检查 git 日志了解最近的工作。
[Tool Use] <bash - git log --oneline -20>
[Assistant] 让我启动开发服务器并验证基本功能。
<启动开发服务器>
<测试基本功能>
[Assistant] 基本功能运行正常。让我查看下一步需要实现什么。
<开始处理新功能>
```

### 智能体失败模式与解决方案

|**问题**|**初始化智能体行为**|**编码智能体行为**|
|---|---|---|
|过早宣布胜利|设置功能列表文件|开始时阅读功能列表，选择单个功能|
|环境留有 bug|初始 git 仓库和进度文件|开始时读进度文件和 git 日志，运行基本测试|
|过早标记功能完成|设置功能列表文件|仔细测试后才标记为通过|
|花时间搞清如何运行应用|编写 [init.sh](http://init.sh) 脚本|开始时阅读 [init.sh](http://init.sh)|

### 未来工作

这项研究展示了长时间运行智能体框架中的一组可能解决方案。仍然存在的开放问题包括：

- 单一通用编码智能体是否最优，还是多智能体架构（如专门的测试智能体、QA 智能体、代码清理智能体）能做得更好

- 将这些发现推广到其他领域（科学研究、金融建模等）

## 📝 英文原文 / English Original

As AI agents become more capable, developers are increasingly asking them to take on complex tasks requiring work that spans hours, or even days. However, getting agents to make consistent progress across multiple context windows remains an open problem.

The core challenge: agents must work in discrete sessions, each new session begins with no memory of what came before.

We developed a two-fold solution: an **initializer agent** that sets up the environment on the first run, and a **coding agent** that makes incremental progress in every session while leaving clear artifacts for the next session. Code examples in the [quickstart](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding).

### The long-running agent problem

Compaction isn't sufficient. Two failure patterns: (1) the agent tries to do too much at once, running out of context mid-implementation; (2) a later agent instance declares the job done prematurely.

### Environment management

**Feature list:** The initializer agent writes a comprehensive file of feature requirements (200+ features for the [claude.ai](http://claude.ai) clone). Features initially marked as "failing." We use JSON format as the model is less likely to inappropriately change JSON files.

**Incremental progress:** The coding agent works on only one feature at a time. It commits progress to git and writes summaries in a progress file.

**Testing:** Claude was prompted to use browser automation tools (Puppeteer MCP) for end-to-end verification.

### Getting up to speed

Every coding agent runs through: (1) `pwd` to see directory, (2) read git logs and progress files, (3) read features list and choose highest-priority unfinished feature.

### Agent failure modes and solutions

|Problem|Initializer Agent|Coding Agent|
|---|---|---|
|Declares victory early|Set up feature list|Read feature list, choose single feature|
|Leaves bugs|Initial git repo and progress file|Read progress, run basic test|
|Marks features done prematurely|Set up feature list|Self-verify after careful testing|
|Figures out how to run app|Write [init.sh](http://init.sh) script|Read [init.sh](http://init.sh)|

### Future work

Open questions include whether specialized agents outperform a single general-purpose agent, and generalizing to other fields like scientific research or financial modeling.
