# 概述

本文档详细说明 CapCut Editor ([www.capcut.com/editor-graphic](http://www.capcut.com/editor-graphic)) 的完整技术架构链路，从用户访问域名开始，到最终服务器响应的全过程。整个链路涵盖了 DNS 解析、CDN 加速、流量管理、负载均衡、容器编排等多个层面。

---

# 一、DNS 解析层

## 1.1 DIG (Domain Information Groper)

**作用**：DNS 查询工具，用于域名解析的第一步

**详细说明**：

- DIG 是用户端或系统发起 DNS 查询的工具

- 当用户访问 `[www.capcut.com/editor-graphic](http://www.capcut.com/editor-graphic)` 时，首先通过 DIG 查询该域名对应的 IP 地址

- 支持查询多种 DNS 记录类型（A、CNAME、MX、TXT 等）

**DNS 记录类型的含义**

DIG 工具支持查询多种 DNS 记录类型:

- **A 记录**:将域名直接解析为 IPv4 地址(如 192.168.1.1)

- **CNAME 记录**:将域名指向另一个域名(别名)

- **MX 记录**:邮件服务器记录

- **TXT 记录**:文本记录,常用于验证和配置

**最终查询到的值**

在你的架构中,当查询 [www.capcut.com/editor-graphic](http://www.capcut.com/editor-graphic) 时:

- DIG 工具会返回一个 **CNAME 记录**

- 这个 CNAME 记录指向 CDN 的域名(比如 [cdn.example.com](http://cdn.example.com))

**DIG 与 CNAME 的关联**

两步的关系是这样的:

1. **第一步(DIG 查询)**:用户发起 DNS 查询,DIG 工具查询域名记录类型

1. **第二步(返回 CNAME)**:DNS 服务器返回 CNAME 记录,告诉客户端这个域名实际指向 CDN 域名

1. **后续**:浏览器会继续查询 CNAME 指向的 CDN 域名的 A 记录,最终获得实际的 IP 地址

举例说明:

`www.capcut.com` (CNAME) → `cdn.example.com` (A记录) → `1.2.3.4` (IP地址)

这种设计的好处是可以灵活切换后端服务而不需要修改原始域名配置。

**关键信息**：

- Disco：暴露给 SRE（Site Reliability Engineering）

- Record：返回 CNAME 或 SKU 类型

---

## 1.2 CNAME (Canonical Name)

**作用**：域名别名记录，实现域名的间接解析

**详细说明**：

- CNAME 是 DNS 记录的一种类型，将一个域名指向另一个域名

- 在本架构中，`[www.capcut.com/editor-graphic](http://www.capcut.com/editor-graphic)` 通过 CNAME 指向 CDN 域名

- 这种设计的好处是可以灵活切换后端服务，而无需修改原始域名的 DNS 配置

**优势**：

- 便于 CDN 切换和灾难恢复

- 统一管理多个域名的解析

- 支持智能 DNS 调度

# 二、内容分发层

## 2.1 CDN (Content Delivery Network)

**作用**：内容分发网络，加速全球用户访问

**详细说明**：

- CDN 是分布在全球各地的边缘节点网络

- 缓存静态资源（图片、CSS、JavaScript、视频等）

- 根据用户地理位置，将请求路由到最近的 CDN 节点

- 减轻源站压力，提升用户访问速度

**核心功能**：

- **CLB**：负载均衡

- **Weibo**：微博相关配置或集成

**特点**：

- 🔥 图标表示这是高流量、高性能的关键节点

- 支持 HTTPS 加密传输

- 具备 DDoS 防护能力

---

## 2.2 回源机制

当 CDN 节点没有缓存或缓存过期时，会回源到源站获取最新内容。回源路径指向下一层的流量管理系统。

# 三、流量管理层

## 3.1 GTM (Global Traffic Management)

**作用**：全局流量管理，智能调度用户请求

**详细说明**：

- GTM 是字节跳动的全局流量管理系统

- 根据地理位置、运营商、服务器健康状态等因素，将流量分配到最优的数据中心

- 支持多地域容灾和负载均衡

**核心能力**：

- 大陆 GTM ➡️ 国内 SKU 调度

- 大陆以外 GTM ➡️ 全球用户调度

**调度策略**：

- 地理位置就近原则

- 服务器负载均衡

- 故障自动切换

---

# 四、网关与负载均衡层

## 4.1 TTGW (TikTok Gateway)

**作用**：API 网关，统一流量入口

**详细说明**：

- TTGW 是字节跳动的统一网关系统

- 负责接收来自 GTM 的流量，进行路由、鉴权、限流等处理

- 将请求转发到后端的 TLB（Toutiao Load Balancer 或类似的负载均衡器）

**核心功能**：

- TLB 聚合：将多个 TLB 实例进行聚合管理

- 支持全量发布

---

## 4.2 TLB Cluster Pod

**作用**：负载均衡集群，分发请求到后端服务

**详细说明**：

- TLB（类似于 LVS 或 NGINX）是四层或七层负载均衡器

- 以 Pod 的形式运行在 Kubernetes 集群中

- 根据负载均衡算法（轮询、加权轮询、最少连接等）将请求分发到后端服务

**关键特性**：

- **by FSM & Cluster**：通过 FSM（Finite State Machine，有限状态机）和集群管理

- 将请求路由到 PSM（具体的服务实例）

- 计算方式：总流量 ÷ TLB 数量

# 五、应用服务层

## 5.1 TCE Pod (TikTok Container Engine Pod)

**作用**：容器化应用运行环境

**详细说明**：

- TCE 是字节跳动的容器编排平台（类似 Kubernetes）

- 所有应用以 Pod 的形式运行

- Pod 是 Kubernetes 中最小的部署单元，包含一个或多个容器

### 5.1.1 Sidecar + Godly-Web-Server

**Sidecar 容器**：

- **作用**：辅助容器，提供基础设施能力

- **功能**：
    
    - 服务网格（Service Mesh）代理
    
    - 日志收集
    
    - 监控指标采集
    
    - 流量拦截和路由
    

**Godly-Web-Server (主程序容器)**：

- **作用**：核心 Web 应用服务器

- **功能**：
    
    - 处理 HTTP/HTTPS 请求
    
    - 业务逻辑处理
    
    - 调用下游服务
    

**组合优势**：

- Sidecar 模式将基础设施逻辑与业务逻辑解耦

- 便于统一管理和升级基础设施组件

### 5.1.2 TCE (容器内组件)

图中显示的 TCE 组件包含两个重要元素：

- **Mesh/Control**：服务网格的控制平面，管理 Sidecar 的配置和路由规则

- **Other Server**：其他辅助服务

## 5.2 渲染与存储层

### 5.2.1 SSR (Server-Side Rendering)

**作用**：服务端渲染，提升首屏加载速度和 SEO

**详细说明**：

- 在服务器端完成页面的 HTML 渲染

- 将完整的 HTML 返回给客户端，减少客户端渲染时间

- 提升搜索引擎爬虫的抓取效果

**适用场景**：

- 需要 SEO 优化的页面

- 首屏性能要求高的应用

- 内容相对静态的页面

### 5.2.2 SSR FaaS (Server-Side Rendering Function as a Service)

**作用**：无服务器的 SSR 解决方案

**详细说明**：

- 将 SSR 功能以 FaaS（函数即服务）的形式提供

- 按需调用，自动扩缩容

- 降低资源成本，提升弹性

**优势**：

- 无需管理服务器

- 自动伸缩

- 按调用次数计费

### 5.2.3 CSR (Client-Side Rendering)

**作用**：客户端渲染，动态交互体验

**详细说明**：

- 服务器返回基础 HTML 和 JavaScript 代码

- 在浏览器端执行 JavaScript，动态生成页面内容

- 适合单页应用（SPA）和高交互性应用

**适用场景**：

- 需要复杂交互的应用

- 后台管理系统

- 不需要 SEO 的应用

### 5.2.4 TOS (TikTok Object Storage)

**作用**：对象存储服务，存储静态资源

**详细说明**：

- TOS 是字节跳动的对象存储系统（类似 AWS S3）

- 存储图片、视频、文档等静态文件

- 提供高可用、高可靠的存储能力

**典型用途**：

- 用户上传的素材

- 生成的视频文件

- 静态资源（图片、CSS、JS）

# 六、完整请求链路流程

## 6.1 用户请求流程

1. **用户发起请求**
    
    - 用户在浏览器输入 `[www.capcut.com/editor-graphic](http://www.capcut.com/editor-graphic)`
    

1. **DNS 解析（DIG → CNAME）**
    
    - 本地 DNS 或递归 DNS 查询域名
    
    - 获取 CNAME 记录，指向 CDN 域名
    

1. **CDN 加速**
    
    - 根据用户地理位置，DNS 返回最近的 CDN 节点 IP
    
    - 用户请求到达 CDN 边缘节点
    
    - 如果 CDN 有缓存，直接返回；否则回源
    

1. **全局流量管理（GTM）**
    
    - CDN 回源请求到达 GTM
    
    - GTM 根据调度策略，选择最优的数据中心
    

1. **网关处理（TTGW）**
    
    - 请求到达 TTGW 网关
    
    - 进行鉴权、限流、路由等处理
    

1. **负载均衡（TLB Cluster）**
    
    - TTGW 将请求转发到 TLB 集群
    
    - TLB 根据负载均衡算法，选择后端 Pod
    

1. **应用处理（TCE Pod）**
    
    - 请求到达 TCE Pod
    
    - Sidecar 拦截请求，进行服务网格处理
    
    - 主程序（Godly-Web-Server）处理业务逻辑
    

1. **渲染与响应**
    
    - **SSR 路径**：服务器渲染完整 HTML，直接返回
    
    - **CSR 路径**：返回基础 HTML + JS，浏览器端渲染
    
    - 如需静态资源，从 TOS 获取
    

1. **响应返回**
    
    - 响应经过 TLB → TTGW → GTM → CDN
    
    - CDN 缓存可缓存内容
    
    - 最终返回给用户

# 七、架构优势

## 7.1 高可用性

- **多层冗余**：DNS、CDN、GTM、TLB 都支持多节点部署

- **自动故障切换**：任何一层出现故障，都能自动切换到备用节点

- **健康检查**：各层都有健康检查机制，及时发现和隔离故障节点

## 7.2 高性能

- **CDN 加速**：静态资源就近分发，减少网络延迟

- **负载均衡**：请求分散到多个服务器，避免单点过载

- **容器化**：快速启动，弹性伸缩

## 7.3 可扩展性

- **水平扩展**：各层都可以通过增加节点来扩展容量

- **微服务架构**：各组件独立部署，互不影响

- **FaaS 支持**：按需调用，自动扩缩容

## 7.4 安全性

- **DDoS 防护**：CDN 层提供基础防护

- **网关鉴权**：TTGW 统一鉴权和限流

- **服务网格**：Sidecar 提供零信任安全模型
