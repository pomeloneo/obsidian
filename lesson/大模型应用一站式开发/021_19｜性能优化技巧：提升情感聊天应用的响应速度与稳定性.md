# 一、性能为何重要？大模型应用的“生死线”

在传统软件开发中，性能通常被视为“锦上添花”的优化项。但在大模型应用中，它已成为决定产品成败的“生死线”。

## 1.1 用户体验的“心理临界点”

研究表明，用户对 AI 响应时间的容忍度极低：

0.1 秒内：感觉是即时响应，体验最佳。

1 秒内：可接受，但已有轻微延迟感。

3 秒以上：开始焦虑，怀疑系统是否卡死。

5 秒以上：多数用户选择放弃或刷新。

而大模型生成文本本身具有自回归特性（逐词生成），即使调用 GPT-4 级别的模型，平均响应时间也常在 1～3 秒之间。若叠加上下文检索、情感分析、安全过滤等流程，总延迟很容易突破 5 秒，严重影响用户体验。影响大模型体验的核心指标如表 19-1 所示。

## 1.2 成本与可扩展性的核心制约

大模型 API 调用按输入 + 输出的 token 数量计费，且响应时间越长，占用连接资源越多。高延迟意味着：

单次请求成本上升。

并发能力下降。

服务器资源利用率降低。

高峰期易出现雪崩效应。

例如，若每个请求平均耗时 4 秒，则单个后端实例每秒最多处理 0.25 个请求；若优化至 1 秒，则吞吐量提升 4 倍，极大降低单位服务成本。

## 1.3 系统稳定性的“多米诺骨牌效应”

大模型应用通常依赖多个外部服务：LLM API、向量数据库、情感分析模型、内容过滤服务等。任何一个环节出现延迟或故障，都可能引发连锁反应：

请求堆积 → 内存溢出 → 服务崩溃。

超时重试 → 流量放大 → 接口限流。

数据库连接池耗尽 → 全系统不可用。

因此，性能优化不仅是提速，更是构建高可用系统的基石。

# 二、大模型应用的典型性能瓶颈分析

在动手优化之前，我们必须先诊断问题。以下是大模型应用中最常见的五大性能瓶颈。

## 2.1 LLM 推理延迟（核心瓶颈）

大语言模型模型的推理过程，大致分为预填充（Prefill）和解码（Decoding）两个阶段。在预填充阶段，提示的多个词元被一起输入给模型，输入的词元数量可能几百或者几千。在解码阶段（图 19-1 红色输入部分），模型每次输入上一次生成的词元，生成下一个词元，循环这个逻辑直到回答结束。

[![](https://static001.geekbang.org/resource/image/18/44/181265cc612b4497f1fdbb4d6e5cce44.png?wh=2680x826)](https://static001.geekbang.org/resource/image/18/44/181265cc612b4497f1fdbb4d6e5cce44.png?wh=2680x826)

Jay Alammar《The Illustrated GPT-2 (Visualizing Transformer Language Models)》” />

由于大模型一般都是采用自回归生成方式，即根据前面的语句预测下一个字的概率，而自回归生成方式会存在以下问题。

1. KV Cache 占用显存大：

每个请求需要维护独立的 KV 缓存；

随着请求数量增加，KV 缓存呈线性增长；

长序列场景下显存压力更大。

2. 解码算法复杂度：

通常需要提供多种解码策略（贪婪解码、采样解码、束搜索）支持；

不同解码策略对内存管理要求不同，例如并行采样和束搜索需要不同的内存管理机制。

3. 动态序列长度：

输入输出长度未知且不固定，要求内存管理系统能够适应不同长度；

解码过程中序列长度持续增长，KV 缓存内存也会相应增加。

同时，现有系统采用静态连续内存分配策略，存在以下三类内存分配问题（图 19-2）。

[![](https://static001.geekbang.org/resource/image/49/89/4973a357faab70e27d5a11a328ddf689.png?wh=2818x426)](https://static001.geekbang.org/resource/image/49/89/4973a357faab70e27d5a11a328ddf689.png?wh=2818x426)

图 19-2 内存让费现象 出自Woosuk Kwon等人2023年发表的《Efficient Memory Management for Large Language Model Serving with PagedAttention》

预留浪费（reserved）：需要为每个请求预留最大可能序列长度的内存空间，导致资源利用率低下。

内部碎片（internal fragmentation）：内存分配效率低导致已分配内存块内部存在未被充分使用的区域，降低内存的整体利用效率。

外部碎片（external fragmentation）：当小内存块无法被有效整合利用时，造成系统可用内存资源的浪费。

LLM 推理性能主要受内存带宽而非计算能力限制，而服务吞吐量则受限于可支持的最大批量大小。

在大模型使用过程的体感如下。

模型越大，生成越慢。

上下文越长，计算量呈平方级增长（自注意力机制）。

高峰期 API 服务端延迟上升。

## 2.2 向量检索的成本与耗时

2022 年 ChatGPT 爆火后，向量数据成为公有云增速最快的数据类型。向量检索很好，但是向量检索最大的问题在于，向量检索的算力需求太高，相比 NoSQL 至少高出了一到两个数量级。甚至，对于一些 AI 笔记类产品来说，他们在向量检索上的开销是 OpenAI API 费用的 2 倍。

与早期服务的搜索、广告场景不同，大模型对应的 RAG 方案，对向量数据库带来了全新挑战，主要包括后面几个方面。

数据量暴增：规模从过去的千万级，直接跃升至数十亿级，需要向量数据库具备海量数据处理。

用户延迟容忍度提升：毕竟大模型本身推理就需要时间，即便向量检索稍慢，用户也能接受；因此，过去向量数据库最在意的召回速率，反而不再重要。

成本敏感度飙升：企业数据量翻倍后，若仍沿用过去的存储和计算方案，成本压力根本扛不住。

为了应对成本压力，向量数据库的底层存储系统，从内存进化到对象存储，其实是行业必然。以 Milvus 为例，向量数据库的演进可以分为三个阶段。

第 1 代（2018-2022 年）：全靠内存撑早期 Milvus 用的索引技术，比如 HNSW、IVF，会把数据放内存里。优点是速度快、查得准，但成本高到离谱，

第 2 代（2022-2024 年）：磁盘救场 Milvus 2.0 正式推出，在此阶段，我们不仅做了分布式，还搞出了 “DiskANN + 自研 Cardinal 索引”。通过 AIO、io_uring 等技术，可以把部分索引从内存卸载到了磁盘当中，成本一下降了 3~5 倍。这也导致我们的 Zilliz Cloud 容量型实例，当时直接卖爆了。

第 3 代（2024 年起）：分层存储时代这一时期，像 TurboPuffer 这些新玩家，直接把索引放到了 S3 里，存储成本降到 $0.33/GB/ 月，又比之前省了 10 倍！但缺点也明显，查冷数据（不常用的数据）要等 500 毫秒～1 秒，召回率也没那么高。

提示：以上信息出自栾小凡《向量检索快比 LLM 还贵？不支持 S3 的向量数据库，迟早要淘汰！》

在向量数据规模急剧膨胀的背景下，向量检索系统将面临严峻的性能挑战：

向量数据库在百万级数据量下，单次检索可能达 200~500ms。

若未合理设置索引或分片，性能急剧下降。

## 2.3 串行调用导致的“瀑布式延迟”

在大模型应用开发中，一个常见的架构误区是将多个处理模块设计为严格的串行调用流程。典型流程：

用户输入 → 情感分析 → 安全过滤 → 向量检索 → 构造Prompt → 调用LLM → 返回结果

这种“前一步完成，后一步启动”的线性执行模式，看似逻辑清晰、易于实现，实则构成了性能优化中最隐蔽也最致命的瓶颈之一——我们称之为瀑布式延迟（Waterfall Latency）。

什么是“瀑布式延迟”？

想象一条瀑布：水流从高处一级一级落下，每一级都必须等上一级流完才能开始。在软件系统中，“瀑布式延迟”正是指多个服务或处理步骤依次等待、逐层推进的执行方式。每个环节的延迟都会叠加到总响应时间上，形成“延迟累积效应”。

即便每个子模块本身性能良好，但一旦串行叠加，整体体验已逼近用户心理容忍的临界点（3 秒）。若任一环节因网络波动、服务降级或数据量增大而变慢，总延迟将迅速突破 5 秒，导致用户流失。

除了显式的代码顺序执行外，还需警惕以下“隐性串行”陷阱。

数据库锁竞争：多个异步任务同时写入同一张表，导致行锁或连接阻塞。

共享资源争用：如共用一个 Redis 连接池，未配置合理超时与重试。

同步阻塞调用混入异步流程：例如在 async 函数中调用了 requests.get() 而非 httpx.AsyncClient()。

上下文传递错误：某个任务本可提前执行，但因等待某个“通知信号”而被延迟。

诊断建议：使用分布式追踪工具（如 OpenTelemetry、Jaeger）绘制调用链路图，可视化识别延迟热点与串行路径。

## 2.4 高并发下的资源竞争

在大模型应用从单机测试走向真实用户环境的过程中，“高并发”是检验系统健壮性的终极试金石。当成千上万的用户同时发起请求时，看似独立的服务模块可能因共享资源的竞争而陷入阻塞、超时甚至雪崩——这种现象被称为资源竞争（Resource Contention）。它不像代码错误那样显眼，却常常是服务不可用的根源。

现代 Web 服务本质上是“资源调度器”，其性能上限由最稀缺的资源决定。在高并发场景下，多个请求试图同时访问同一类有限资源，导致后面这些情况。

请求排队等待。

等待时间超过用户容忍阈值。

连锁反应引发服务降级或崩溃。

我们来剖析一下典型竞争场景。

### (1) 数据库连接池耗尽

大多数应用使用 ORM（如 SQLAlchemy）配合连接池管理数据库连接。默认配置通常仅支持几十个连接。当突发流量到来时，所有请求争抢连接：

db_session = SessionLocal()

一旦连接池满，后续请求将排队等待，形成“请求堆积 → 延迟上升 → 超时重试 → 更多请求涌入”的恶性循环。

案例：某 AI 客服系统在促销活动期间，每秒新增 300 个对话请求，但 PostgreSQL 连接池仅配置为 50。结果大量请求卡在数据库层，平均响应时间从 800ms 飙升至 12s，最终触发前端网关熔断。

### (2) Redis 缓存连接风暴

Redis 常用于存储会话状态、限流计数器、缓存结果等。若未合理配置连接池或超时策略，多个异步任务同时操作 Redis 会导致：

单线程 Redis 成为瓶颈。

客户端连接积压。

TCP backlog 溢出。

尤其在缓存穿透或缓存击穿场景下，大量请求绕过缓存直接打向后端，进一步加剧压力。

### (3) GPU 显存资源争抢（LLM 推理特有）

对于自建 LLM 推理服务（如部署 Llama 3），GPU 显存是最关键的稀缺资源。每个推理请求需分配 KV Cache 空间，且显存不能像内存一样交换到磁盘。当并发请求数超过 GPU 容量时：

新请求必须等待旧请求释放显存。

出现“显存碎片化”，即使总空闲显存足够，也无法满足新请求。

导致 P99 延迟急剧上升。

例如：一张 A100（80GB）可支持约 20 个并发推理任务（取决于上下文长度），若不加控制，第 21 个请求将被迫排队或失败。

### (4) 网络 IO 与带宽瓶颈

大模型输出动辄数千 token，若采用流式传输（Streaming），每个活跃连接将持续占用网络通道。假设：

平均每个响应大小为 5KB。

每秒生成 10 个 token。

同时有 1000 个用户正在接收流式回复。

则所需下行带宽 ≈ 5KB × 10 × 1000 = 50MB/s（400Mbps）。若服务器出口带宽仅为 100Mbps，则必然出现拥塞丢包。

## 2.5 前端交互卡顿

即便后端优化得再完美，如果前端无法流畅呈现 AI 的回应，用户的感知依然是“卡顿”“慢”“不智能”。前端作为人机交互的最终界面，承担着将技术能力转化为良好体验的关键使命。

我们来看看卡顿的三大根源。

### (1) 长轮询（Long Polling）滥用

部分老旧架构仍采用长轮询方式获取 AI 回复：

async function pollForResponse() {

```jsx
while (true) {
const res = await fetch('/check-response');
if (res.data) break;
await sleep(500);
}
}
```

这种方式存在严重问题：

高延迟感知：用户提问后至少等待一个轮询周期才能看到反馈。

服务器压力大：每个未完成的对话维持一个 HTTP 连接。

移动端耗电严重：频繁唤醒设备。

在 500ms 轮询间隔下，平均额外增加 250ms 延迟，完全破坏“即时响应”的心理预期。

### (2) WebSocket 未优化：消息洪流冲击 UI 线程

虽然 WebSocket 支持全双工通信，但如果后端以极高频率发送 token（如每 10ms 一个），前端处理不当会导致后面这些问题。

UI 线程被频繁打断。

浏览器重排（reflow）和重绘（repaint）过于密集。

文本闪烁、滚动失灵、页面卡死。

此外，若未实现心跳保活机制，在弱网环境下连接易断开，造成“假死”现象。

### (3) 缺乏流式渲染（Streaming Rendering）

许多前端应用选择等待 LLM 完整输出后再一次性更新 DOM：

```jsx
const response = await fetch('/chat').then(r => r.text());
setMessage(response);
```

这使得用户在前几秒内看到的是空白屏幕，极易产生“是否发送成功？”的焦虑感。

研究表明，用户对响应速度的主观感受不仅取决于绝对时间，更受反馈节奏影响，如表 19-2 所示。

# 三、性能优化的四大核心策略

针对上述瓶颈，我们提出四大优化策略：并行化、缓存化、流式化、降级化。

## 3.1 策略一：并行化处理——打破“瀑布流”延迟

目标：将串行流程尽可能并行执行，减少总耗时。

优化前（串行）：

[情感分析] → [安全过滤] → [向量检索] → [调用LLM] → [返回]

耗时：200ms + 100ms + 300ms + 2000ms = 2.6s

优化后（并行）：

同时启动：

- [情感分析]

- [安全过滤]

- [向量检索]

↓

等待所有完成（取最慢者，约300ms）

↓

[构造Prompt + 调用LLM]

↓

[返回]

总耗时 ≈ 300ms + 2000ms = 2.3s（节省300ms）

实现方式（Python 示例）：

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor
async def process_input_parallel(user_input):
loop = asyncio.get_event_loop()
with ThreadPoolExecutor() as pool:
emotion_task = loop.run_in_executor(pool, analyze_emotion, user_input)
safety_task = loop.run_in_executor(pool, check_safety, user_input)
retrieval_task = loop.run_in_executor(pool, retrieve_memory, user_input)
emotion, safety, memory = await asyncio.gather(emotion_task, safety_task, retrieval_task)
prompt = build_prompt(user_input, emotion, memory)
response = await call_llm_streaming(prompt)
return response
```

这里提示一下，使用 asyncio + ThreadPoolExecutor 可高效实现 I/O 密集型任务的并行化。

进一步优化：早期中断机制

当并行任务中某个任务提前返回关键结果（如安全过滤未通过），可以立即中断其他任务，避免资源浪费：
```text
async def process_input_parallel_optimized(user_input):
loop = asyncio.get_event_loop()
tasks = {}
with ThreadPoolExecutor() as pool:
tasks['safety'] = loop.run_in_executor(pool, check_safety, user_input)
tasks['emotion'] = loop.run_in_executor(pool, analyze_emotion, user_input)
tasks['retrieval'] = loop.run_in_executor(pool, retrieve_memory, user_input)
safety_result = await tasks['safety']
```
```text
if not safety_result['passed']:
tasks['emotion'].cancel()
tasks['retrieval'].cancel()
return {"error": "内容不符合安全规范"}
emotion, memory = await asyncio.gather(tasks['emotion'], tasks['retrieval'])
prompt = build_prompt(user_input, emotion, memory)
response = await call_llm_streaming(prompt)
return response
```

这样做的优势是，对于安全过滤这类“一票否决”的任务，一旦返回未通过结果，可以立即取消向量检索等耗时操作，节省计算资源并减少响应延迟。适用于有明确失败条件，而且执行时间较短的任务。

## 3.2 策略二：缓存机制——避免重复计算

策略二是缓存机制，目标是对高成本、低变化的操作结果进行缓存，显著降低延迟与成本。

### 3.2.1 向量检索结果缓存

场景：用户重复提问类似内容（如“我今天很难过”）；

方案：将用户输入的 embedding 哈希值作为 key，缓存检索结果；

工具：Redis、Memcached。

```python
import hashlib
import redis
r = redis.Redis()
def cached_retrieve(user_input):
key = "memory:" + hashlib.md5(user_input.encode()).hexdigest()
cached = r.get(key)
if cached:
return json.loads(cached)
result = vector_db.search(get_embedding(user_input))
r.setex(key, 3600, json.dumps(result))
return result
```

### 3.2.2 LLM 响应缓存（谨慎使用）

LLM 的响应缓存建议谨慎使用，仅适用于事实性问答（如知识库查询），不适用于情感对话（每次上下文不同）。可缓存“常见问题”的标准回复。

### 3.2.3 Prompt 模板缓存

将动态构造的 Prompt 模板预编译并缓存，避免每次字符串拼接。

## 3.3 策略三：流式响应（Streaming）——提升感知速度

此外我们还有流式响应的策略，目标是实现“边生成边看到”，显著降低用户的主观延迟感。

虽然 LLM 生成仍需时间，但通过流式传输（chunk by chunk），用户可在 100ms 内看到第一个字，极大改善体验。

FastAPI + SSE（Server-Sent Events）示例：

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
app = FastAPI()
async def generate_stream(prompt):
for token in llm_client.stream(prompt):
yield f"data: {token}\n\n"
await asyncio.sleep(0.01)
@app.post("/chat")
async def chat(user_input: str):
return StreamingResponse(
generate_stream(build_prompt(user_input)),
media_type="text/event-stream"
)
```

前端 React 处理流式数据：

```jsx
const response = await fetch('/chat', { method: 'POST', body: input });
const reader = response.body.getReader();
let text = '';
while (true) {
const { done, value } = await reader.read();
if (done) break;
text += new TextDecoder().decode(value);
setResponse(text);
}
```

效果：用户感知延迟从“等待全部生成”变为“即时反馈”，体验提升显著。
