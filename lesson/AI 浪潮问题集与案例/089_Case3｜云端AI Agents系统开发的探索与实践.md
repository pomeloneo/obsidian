当前，AI Agent 的开发呈现爆发式增长，机遇与挑战并存。在大模型逐步展现通用智能特性的背景下，AI Agents 已在客服、软件开发、财务分析、科研辅助等领域崭露头角，展现出强大的场景扩展潜力。与此同时，三大技术瓶颈亟待突破：算力成本指数级攀升、系统行为不可控性加剧、长周期复杂任务成功率不足。在 InfoQ 举办的 QCon 全球软件开发大会上，亚马逊 Agentic AI Science 团队主任科学家章毅分享了“云端 AI Agents 系统开发的探索与实践”，他结合金融分析、电商平台等典型场景，深入解析了三大破局方案——基于多智能体协作架构、混合编排系统、以及集成多种验证机制的实时监控层，并系统性展望 AI Agents 在云计算范式下的演进路径与商业落地可行性。

## Al Agent 应用现状和核心技术挑战

对于 Agent 概念的提出，其实早于我们在机器学习领域对 Agent 的定义。一个比较重要的工作是 Minsky 在 1986 年出版的一本书，名为《心智社会》（Society of Mind）。在这本书中，Minsky 作为一名计算科学家，提出了他对人类心智的理解。他提出了智能体（Agent）的概念，认为所谓的智能体本身可能是一些非常简单、非常基础的，甚至并不很智能的过程。但通过这些不同的智能体作为基础的构建块相互连接、相互协作，可以构造出一个庞大的“社会”形式，从而达到更高层次的智能表现。这个概念非常有趣且具有深度内涵。不过，与我们现在讨论的基于大型语言模型（LLM）的智能体相比，它其实存在一些不完全兼容的地方。他提出的原始智能体定义相对简单，不需要一个庞大的模型，比如几百亿参数的模型来实现。它可能是一个相对简化的过程，也不一定在单个智能体层面体现出很高的智能水平。但它的重要之处在于，它可以通过叠加和结构化的社会体系，达到更高层次的智能。回到这个定义，当时提出这个概念时，这本书并没有给出一个很明确的数学化、形式化的模型。因此，对于如何实现这样一个设想，其实有很大的解读空间。

快进到相对比较近期的机器学习领域，尤其是在强化学习的背景下，Agent 的定义被进一步数学化了。在强化学习的经典定义中，Agent 通常被认为是一个交互系统，它与所处的环境进行交互，这种交互本身相对简单。Agent 可以采取行动，执行一些操作。这些操作在动作空间中可能是离散的，也可能是连续的。同样，状态空间也有类似的情况。在某些情况下，状态空间可以是预定义的，比如在预定的工作流程中，你可以预定义一些具体的状态，并针对这些 状态提供更好的控制和合规性检查。但在其他情况下，例如像基于大型语言模型的 Agent，它本身隐式地建模了 Agent 的空间。这种模型被定义为一种马尔可夫决策过程，具有一定的灵活性。

在最初提出基于强化学习的 Agent 概念时，它并没有像现在这样流行。主要原因可能有几点：一是当时缺乏强大的预训练模型。要实现基于强化学习的 Agent，需要处理大量的训练复杂度，而强化学习算法本身的运行也不像传统的监督学习那样稳定。因此，从可实现性的角度来看，它面临着很大的挑战。不过，它的内部主要学习目标已经比较明确地定义了，即最大化累积预期奖励。根据这个目标来优化策略函数 π，这个决策函数会根据当前 Agent 和环境的状态来决定 Agent 接下来需要采取的动作。

随着大语言模型的兴起，其展现出强大的泛化能力。对于许多新兴任务，即使没有现成的训练数据，通过 LLM 的泛化能力与提示词（prompt）结合，可以大大简化 Agent 系统中策略模型的实现难度。无需真正进行强化学习训练，我们可以在短时间内（如 5 分钟到 20 分钟）构建出 Agent 原型。

除了大语言模型本身，还有一些其他重要的组成部分，我们将其称为编排逻辑。编排逻辑可以根据其类别分为两种。一种是静态编排逻辑，例如常见的工作流或像 LangChain、LangGraph 这样的实现。在 AWS 上，可以使用微服务（如 AWS Step Functions），方便地手工定义这些编排逻辑。另一种情况是将编排逻辑的制定也交给大语言模型来自主完成，让它自主编排执行过程。虽然存在不同的变体，但总体而言，这些都可以视为基于反应式模型的变体。

在这种情况下，LLM 根据自身的推理结果决定如何调用工具，并根据工具调用的结果构建观察结果，进而决定下一步的动作（action）。总体而言，这与之前提到的强化学习 Agent 的设定基本兼容，唯一的变化是由于 LLM 的出现，构建策略模型的难度大幅降低。

除了 Agent，现在还在不断发展的一个概念是 GenAI Agent 的认知架构。这大致定义了 Agent 除了核心部分（通常由大模型驱动，并可能配备一组编排逻辑与大模型协同工作）之外，还包含哪些组件。一般来说，大家比较认可的组件包括长期记忆和短期记忆（如对话记录或近期有限轮对话记录，以及近期的函数调用结果，这些都可以包含在输入上下文中）。然而，对于长期记忆的实现方式，目前在基本的 LLM Agent 实现中仍存在很大争议。例如，是将这些记忆存储在数据库中，以记录形式访问，还是以其他方式修剪上下文，存在不同的实现方式。但无论如何，我们认为可能需要多种不同的记忆保留机制的组合来实现 Agent 的长期记忆组件。

规划也是如此。一方面，可以利用 LLM 本身来决定下一步的执行任务步骤；但在某些情况下，针对特定应用领域，可能已经存在一些成熟的规划算法。例如，在我们针对 Amazon Q 的代码迁移工作中，需要将 IBM 主机代码转换为 Java 代码。在这种情况下，规划更多地是从现有的知识库中检索已有的工作流程和步骤，而不是直接交给 LLM 去规划。因为 LLM 的预训练数据中可能无法涵盖很多特定领域的规划数据，它可能缺乏这些特定规划的能力。

交互和通信也是非常重要的一部分，因为 Agent 不仅需要与人交互，还需要与环境交互。人应该被视为 Agent 与环境交互中的一个组成部分，但环境中还可能包含其他 Agent、多个用户、不同的 API 和知识源（有些是本地的，有些是在线的），环境是非常复杂的。目前，大家通常通过像 MCP 这样的标准来初步标准化基础模型与工具之间的接口。但我们认为，随着类似协议的演变，它将逐渐涵盖更广泛的 Agent 与环境交互的标准化。

在 2024 年和 2025 年，AI Agent 领域已经出现了一些比较成功的应用案例。例如，在软件开发领域，有像 Cursor 这样的工具。在亚马逊，我们有 Amazon Q，它在内部得到了很好的应用。Salesforce 也构建了一个名为 Agentforce 的平台，这是一个针对特定领域的 AI Agent 平台。亚马逊最近还上线了一个名为 Nova Act 的产品，它是一个针对计算机用户交互的 AI Agent，可以帮助用户与浏览器进行交互，自动化一些在线任务。

尽管目前有一些早期的成功应用，但从整体来看，AI Agent 的开发仍面临许多挑战。许多原型系统虽然能够完成一些简单任务，但在解决复杂问题时存在困难，导致它们长期停留在原型阶段。据我们基于 AWS 的 Agent 开发平台观察，每周有数千个新的 Agent 原型被开发出来，但很少有应用能够真正扩展到可部署的阶段。

经过分析和与客户的交流，我们发现主要有三个问题。首先，人们对 AI Agent 的能力期望与当前任务完成的成功率之间存在不匹配，尤其是在复杂任务（如多步骤任务）中，成功率通常低于 50%，而客户期望至少达到 80% 甚至 90%。其次，AI Agent 的行为缺乏一致性。

即使对于重复的任务，Agent 的执行步骤也可能存在不确定性，这在需要严格遵循流程的领域（如医疗、金融等）是一个重大问题。最后，算力成本较高。与传统的交互系统（如 AWS Lex）相比，基于大语言模型的 AI Agent 的计算成本和响应延迟都显著增加，但效果提升并不成比例。尽管如此，多智能体协作被认为是一种可能有效的新设计范式，尤其是在提高准确性和可控性方面。

### 基于多智能体合作的技术路线

多智能体（Multi-agent）并不是一个很新的概念，它其实是一个非常直观的概念。多智能体系统的定义中就包含了多个智能体的成分，这些智能体之间不是独立的，而是相互连接的，它们可能合作，也可能竞争，以不同的模式进行互动。不同的多智能体系统可以分为合作型多智能体系统或竞争型多智能体系统，还有混合型的系统。这些系统的理论基础主要在博弈论中。如果智能体之间是正和游戏，那么它基本上是一个多智能体协作系统；如果它们之间是零和游戏，说明它们之间存在竞争关系，那么它是一个竞争型多智能体系统。这些分类并不是完全互斥的，有时设计可以是协作团队与多团队竞争并存，具体应用中存在许多变体。

在具体应用层面，比较典型的两类合作模式是层次式合作和平级合作。层次式合作类似于各种组织机构中的人类组织结构，通常有一个层级结构，例如“你的上级是谁？”这种单线报告关系，形成一个树状结构。我们发现这种树状的层次化组织结构是一种有效的协作模式。与之相对的是平级合作，这种情况下，团队成员之间没有上下级关系，以一种更民主、更平等的形式进行合作，这种合作模式对于某些任务，尤其是增强合作团队的健壮性，是非常有帮助的。

一个比较典型的应用是多智能体辩论，它通常是一种平级合作的形式，每个智能体都可以挑战其他智能体的观点，这种方式可以增强系统的决策健壮性。在决策形式上，层次式合作中通常采用中心化决策，由某个中心节点做出最终决策；而平级合作中通常采用去中心化决策机制，例如可以让智能体对某个具体决策进行投票，投票方式可以是多数投票或一致通过投票等形式。

这些合作机制各有优缺点，是否适用最终取决于具体的应用场景。经过大量案例分析后，我们发现层次化合作在企业级客户应用场景中具有较广泛的适用性。然而，层次化合作也存在一些已知问题。如果简单地实现层次化合作，主要问题是层次结构会导致决策和响应的延迟大幅增加。例如，如果合作网络有三层结构，延迟可能会增加 3 倍甚至 6 倍。因为在这种情况下，信息需要从顶层智能体传递到底层子智能体，再根据子智能体的结果反馈回顶层进行汇总。这不仅增加了推理代码的数量，还导致响应延迟表现不佳。

针对这些问题，我们在 Bedrock Agents 平台上对多智能体协作进行了进一步优化。首先，我们引入了并行化的智能体通信机制。当一个监督智能体决定将任务分配给三个子智能体处理时，并不是同步或串联地执行子任务，而是并行处理。在这种情况下，从主智能体到子智能体的通信被简化，子智能体只需要知道它需要执行的子任务是什么。所有汇总和总结的工作仅由监督智能体层面的大模型完成，从而大大简化了子智能体之间的通信。

第二个优化是引入了负载引用机制。例如，在一个软件开发任务中，一个子智能体负责编写代码，另一个子智能体负责测试和调试代码。第一个子智能体生成的代码（可能是几十个、上百个甚至上千个 token 的 Python 代码）通常需要传递给监督智能体，然后再传递给另一个子智能体。但在这个过程中，如果任务类似于团队成员完成代码后需要另一个成员进行代码审查，那么并不需要通过团队经理（监督智能体），因为经理在这种情况下并不直接参与审查。因此，我们设计了一个机制，允许生成的内容绕过监督智能体的推理过程，直接传递给测试子智能体。这样可以避免不必要的 token 生成，从而减少推理负载。

第三个优化是动态智能体调度机制。很多时候，用户提交的任务在交互回合中的复杂度是有限的。虽然有些任务可能需要监督智能体进行分解并分配给不同的子智能体，但许多任务本质上非常简单，只需要其中一个子智能体就能解决。在这种情况下，完全可以绕过完整的监督智能体大模型推理，通过一个小模型直接将任务分配给合适的子智能体。同样，当子智能体完成任务后，由于任务本身简单且没有进一步分解，子智能体的输出可以直接返回给用户，从而在返回路径上也绕过了监督智能体的推理负载。

在引入这些优化机制后，我们进行了一些评测。我们使用了一个匿名的开源多智能体框架，并收集了一些数据集来进行测试。在我们的 Bedrock Agents 中实现的多智能体协作，无论是在任务成功率还是效率方面，都明显优于开源框架。这种优势在三个不同的商业领域的数据集上都有显著体现。我想强调的一点是，通过深度优化，我们可以在关键的开发范式上进行改进。例如，多智能体协作可以通过深度优化实现更高效的效果，而这些优化在云端的实现可以显著降低面向开发者的复杂性。

## 应用实例分析

### 电商客服流程自动化系统 MARCO

亚马逊是一个大型电商平台，目前在亚马逊上从事零售业务的卖家大约有 1,000 万，销售的商品大约有 6 亿 种。对于如此庞大的卖家群体，存在大量的客服需求。传统的做法是依靠所谓的客服代表或内部称为领域专家的人员来处理卖家的问题。例如，卖家可能会问：“我的商品已经上架，但为什么客户无法点击‘一键购买’？” 这个功能可能会受到多种条件限制。例如，如果内部检测机制发现卖家销售该产品无法盈利，且存在明显的亏损，系统会自动禁用“一键购买”按钮。卖家通常不知道为什么会发生这种情况，因此需要联系客服来了解和解决问题。

在这种情况下，传统客服需要查阅内部不同的 SOP，调查可能的原因，进行根源分析，并调用工具来解决问题，例如调整零售价格或改变运输方式。这是一个需要人工参与的复杂过程，且传统上是一个高度人工驱动的过程。

现在的情况有所不同。例如，卖家想了解他的产品编号为 B1234xxx 的商品是否可“一键购买”。他询问了一个名为 Marco 的系统。这个系统会调用相关的 Agent 数据库，根据可购买性 的 SOP 进一步调查。在这个过程中，系统需要保持接近实时的响应时间，因为用户是通过聊天界面与 Marco 系统进行交互的。在调查过程中，用户可能会提出新问题，例如询问“CRAP”是什么意思。系统需要并行处理这些问题，从知识库或索引存储中获取答案。实际上，“CRAP”是“Cannot Realize a Profit”的缩写，这是一种保护卖家的机制。

这个系统的核心是基于多智能体实现的自动化的客服系统。最上层是一个名为 Marco Base Agent 的基础协调监督智能体。它下面针对每一个不同的 SOP 都有一个具体的智能体实现，有些 SOP 在解决问题时可能会调用另一个 SOP 的智能体，形成一个级联的层次化多智能体网络。

这种实现可以满足许多需求，但在开发过程中也遇到了一些问题。大约一年前，当时主流的模型是 Claude 2.1 版本。虽然它很多时候可以正确调用工具，但无法保证完全不出错。例如，它可能会在调用函数时遗漏参数，或者错误地生成函数名称。为了解决这些问题，我们引入了不同的护栏机制，对 LLM 的输出进行进一步验证和校验。有些护栏机制是基于规则的系统，例如在生成函数调用时，我们会根据工具的 schema 进行语法验证和类型检查。还有一些验证机制可能需要使用机器学习模型甚至 LLM 来实现，尤其是针对幻觉问题的验证。验证是第一步，发现问题后，第二步是将这些问题反馈给智能体。我们通过模型反思的方式实现这一点。针对每一类护栏机制发现的错误，我们都有一个特定的 LLM 提示词，将错误反馈给智能体，给智能体一个重试的机会。我们发现，设置不超过两次重试，也能显著提升系统的正确性，而且对延迟的影响不大。这是因为延迟是按交互轮次计算的，而我们通过并行和异步设计，确保了响应延迟没有增加。

我们对反思机制进行了比较研究。如果采用简单粗暴的方式，即发现错误后直接让模型重新生成内容，这种方法往往效果不佳。即使重试多次（比如四次），仍然会有许多错误无法修复。然而，如果我们提供自然语言形式的反馈，并基于这种反馈进行反思，通常在一到两次反思和重试之后，大部分错误都能得到解决。

## 云端 Agent 技术前瞻

我们发现 AI Agent 的开发过程中，许多 Agent 原型仍然停留在所谓的原型开发阶段。但我们也意识到，随着这些问题在未来得到解决，更多 Agent 被部署后，部署监控和优化方面的问题将逐渐凸显。

总体而言，目前开发者工具的缺失不仅体现在如何帮助开发者构建 Agent，还体现在部署环节。例如，如何安全地部署这些 Agent？是通过 Web 浏览器，还是通过虚拟机（Virtual machine）？此外，如何监控它们的性能和安全性，并进一步实现持续迭代学习？目前这些工具都尚未完全成型。这不仅适用于像 AWS 这样的云服务提供商，开源社区也面临同样的问题，这些领域都为未来发展留下了巨大的空间。

我所说的云端 Agent 服务技术栈中，目前存在的差距是基础算力服务。例如，AWS 提供了 EC2 以及多种计算实例类型，微软和谷歌等公司，以及许多国内大厂也提供类似的服务。在基础算力服务之上，需要的是基本的生成式 AI 推理托管服务。在 AWS 上，我们的产品主要是 AWS Bedrock，也包括 SageMaker，但类似的服务在其他云平台上也已经存在。然而，要构建垂直 AI agent 应用，目前还存在一个巨大的中间缺失层。我认为，未来半年、一年甚至两年内，许多公司、初创企业和大型企业都会投入大量资金来开发这个中间层，我们可以将其称为 AI agent 的基础组件。

最后总结一下，我主要讨论了 LLM 赋能的 Agent 的起源，以及它应用的现状。但目前 AI Agent 系统开发中存在几个主要的痛点，多智能体协作可能是一种新的设计范式，但仅靠它还不足以解决所有问题。我们通过实例分析发现，即使在采用多智能体协作的情况下，有效的护栏机制仍然至关重要。最后，我认为很快会有一批新的云端 Agent 基础组件出现，为未来 Agent 的普及奠定基础。
