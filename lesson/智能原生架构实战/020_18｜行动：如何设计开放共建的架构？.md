# 18｜行动：如何设计开放共建的架构？

原文链接：https://time.geekbang.org/column/article/911589

[![](https://static001.geekbang.org/resource/image/03/8c/03ee673a65390d234352ca618852098c.png)](https://static001.geekbang.org/resource/image/03/8c/03ee673a65390d234352ca618852098c.png)

你好，我是灵犀。

前些天，蔚来发布了全新车型乐道 L90，其中最具争议性的设计，当属“前置箱”的引入。尽管大家对其褒贬不一，但不可否认的是，这一设计是从用户视角出发，所做出的一种创新尝试。

尤其是在“钓鱼”这一生活化场景中，蔚来围绕“前置箱”讲述了一个极具画面感和代入感的故事，让公众能够迅速理解并产生共鸣。

我们切回架构的主题，在之前关于“业务响应力”的课程中，我们曾经说过，业务人员深度参与 IT 系统建设，已不再是“可选项”，而是未来的“必选项”。这意味着，架构的直接服务对象，正在从“技术”向“业务”延伸。

换句话说，“从用户出发”的思维方式，未来不仅要体现在产品设计上，也要直接体现在架构中。

那么，正如蔚来通过结合场景化讲出了一个令人信服的故事一样，架构也必须能够讲出一个业务听得懂、用得上的“故事”。这不仅是沟通的桥梁，更是价值交付的关键支点。

那么，今天我们就来探讨这一话题——这种开放共建的架构，应该如何设计？

## 传统多层架构问题

为了更清晰地把握未来架构演进的方向，我们首先需要理解传统应用架构所面临的局限性。

传统的应用架构通常采用多层级的分层模型，比如常见的“展示层、应用层、服务层、数据层”结构。这种架构设计在传统的“IT 主导、业务被动使用”的开发模式下，具有一定的合理性，能够实现模块解耦、便于维护与扩展。

然而，随着企业对“业务与 IT 共建”模式的不断探索和推动，传统架构逐渐暴露出一些核心问题，影响了其在新时代下的适用性。我们具体来看看都有哪些不便。

[![](https://static001.geekbang.org/resource/image/57/02/57febd91449637b5094dea1f1e416902.jpg?wh=4000x2250)](https://static001.geekbang.org/resource/image/57/02/57febd91449637b5094dea1f1e416902.jpg?wh=4000x2250)

一、层次繁多，可理解性差

传统架构的设计逻辑多从技术视角出发，对业务人员而言缺乏直观性。各层级之间的职责划分尽管存在，但业务人员难以理解其背后的逻辑关系。例如，一个功能应该放在“应用层”还是“服务层”？某个数据处理逻辑应该放在哪个层级？

这类问题对业务人员而言缺乏明确的判断标准，导致他们在参与系统共建时，难以准确理解系统结构，影响沟通效率与参与深度。

二、约束严格，灵活性受限

传统架构中，层间调用关系受到严格限制，如“只能上层调用下层”“不能跨层调用”等。虽然这种设计在一定程度上保障了系统的稳定性，但也带来了显著的局限性。

一是访问路径冗长。一个简单的业务数据获取，往往需要经过多个中间层处理，响应效率低。尤其在未来微卡片、智能体等轻应用形态下，对数据的直接暴露和快速响应提出了更高要求。

二是变更路径复杂。一次业务规则的调整，可能需要修改多个层级的代码，导致开发成本高、响应周期长。

这些问题不解决，即使业务人员参与进来，面对复杂的调用链条，也难以真正发挥价值。

三、服务封闭，不利于共建

传统架构本质上是一种“IT 主导、业务被动使用”的封闭体系。它对外暴露的接口数量有限，且接口设计多从技术视角或页面驱动出发，缺乏对业务能力的抽象与开放。

这种封闭性限制了业务能力的复用和共享，使得跨团队、跨系统的协作变得困难，也不利于构建一个开放、灵活、可共创的业务生态。

简而言之，传统应用架构完全以技术视角为核心，在系统模块解耦、可维护性等方面确实具有优势。然而，在当前业务与 IT 共建的趋势下，这种架构缺乏对业务场景的自然映射，在 API 接口设计、数据模型表达以及业务逻辑组织等方面，难以与业务人员的思维方式和使用习惯相匹配。

## 未来架构如何演进

为应对上述挑战，我们需要对传统应用架构进行系统性重构，目标是打造一个面向业务开放、易于业务理解、支持业务深度参与，并能快速响应业务变化的新一代架构体系，如下图所示。

[![](https://static001.geekbang.org/resource/image/81/1f/810380443d570e5441959e5311e87b1f.jpg?wh=4583x4393)](https://static001.geekbang.org/resource/image/81/1f/810380443d570e5441959e5311e87b1f.jpg?wh=4583x4393)

在新的应用架构中，我们对整体结构进行了大幅简化，聚焦于两个核心层级：中心平台与外围应用。该架构采用了完全扁平化的设计理念，摒弃了传统架构中复杂的多层级结构，使系统逻辑更清晰、结构更直观，降低了业务人员的理解与参与成本。

在这一架构中，中心平台是整个企业的核心能力中枢，承载着企业核心的业务、数据与技术等关键能力。平台内部的服务均可对外开放，支持按需调用和灵活集成，支撑对业务需求的快速响应。

外围应用则聚焦于具体的业务场景，负责能力的快速组合与落地。其设计目标是实现“可插拔、可组合、可替换”。它可以灵活接入中心平台，根据业务需求快速构建、部署，并随着业务变化进行灵活调整或替换，真正实现“即插即用”。

简而言之，中心平台相当于企业级的“操作系统”，而外围应用则像一个个灵活的“U 盘”，可以便捷地与平台连接，可根据需要随时插入或拔出。这种松耦合的结构，极大提升了系统的适应性与扩展性。

为了能够更直观地将这一架构理念传达给业务团队，我们也将新架构命名为“八爪鱼架构”。中心平台如同八爪鱼的“大脑”，提供统一的能力支撑与决策中枢。而外围应用就像一只只灵活的“触角”，深入各个业务场景中，实现科技对业务的深度赋能。

接下来，我们将围绕中心平台与外围应用，探讨它们在新架构体系下的建设原则，帮助你进一步理解两者的功能定位和实现思路。

## 中心平台建设原则

首先，我们先从整体视角看一下中心平台在建设过程中应遵循的核心原则。更具体的内容，我们将在下节课再展开说明。

[![](https://static001.geekbang.org/resource/image/b8/95/b86c1303b98d4996358b589923edd295.jpg?wh=4000x2250)](https://static001.geekbang.org/resource/image/b8/95/b86c1303b98d4996358b589923edd295.jpg?wh=4000x2250)

原则一：服务化

在新的架构下，中心平台的对外能力应全面服务化，以支持外围应用按需调用与灵活集成。服务化能力可以通过多种交付方式实现，主要包括以下四种。

1. API 接口方式

平台中的应用应基于业务领域建模，围绕业务语义设计接口，避免传统“页面驱动”的接口设计方式。

2. 事件驱动方式

在进行领域建模时，需识别并定义关键的领域事件，并通过事件中心对外暴露。外围应用可根据业务需要主动订阅并响应事件，从而实现对业务变化的实时感知与联动处理，提升系统的动态响应能力。

3. 数据或文件方式

除了标准服务接口，平台还需支持面向数据的能力开放。例如，在 DataAgent 中，根据用户权限，开放结构化数据与非结构化文件，满足多样化的数据消费场景。

4. 平台化开发方式

中心平台应通过平台化手段，尽可能降低开发门槛。例如，对智能体、微卡片等轻量级应用的开发中，可提供低代码或零代码开发平台，让业务人员能够直接在平台上创建与配置。这些应用在开发完成后，可像“插件”一样嵌入平台运行，实现能力的快速组装与灵活部署。

原则二：一体化

尽管中心平台是由多个能力域和大量服务构成，但在规划与建设过程中，必须始终将其视为一个统一的整体产品，进行统筹设计与协同开发。具体而言，一体化原则主要体现在以下几个方面。

1.  统一领域建模

从整体业务视角出发，对平台涉及的各个业务域进行统一建模，尤其关注跨领域业务场景的协同逻辑。通过统一的领域设计，确保业务逻辑的一致性与服务间的高效协同，避免业务功能的重复建设或逻辑割裂。

2. 统一数据模型

建立统一的数据架构和数据标准，打通各服务之间的数据壁垒，实现数据的共享与互通，以保障数据的一致性、准确性和可复用性，为数据驱动和数据分析提供支撑。

3. 统一产品规范

中心平台的功能设计应具备高度的可配置性与灵活性，以满足多样化、多类型业务场景的需求。可借鉴服务网格中“控制面”与“数据面”分离的设计理念，构建统一、标准化的配置管理体系，提升平台的扩展能力与运维效率。

原则三：运营化

中心平台不应仅仅被视为一个承载能力的“静态容器”，而应被重新定义为一个具备持续运营能力的“活性产品”。因此，中心平台建设的核心目标，不只是功能的实现，更在于其在真实业务场景中的价值产出、能力演化以及与业务发展的动态适配。

围绕这一目标，我们提出了一个核心理念——平台的价值必须可度量。平台与外部应用之间的每一次调用、每一条数据流动、每一个服务接口的连接，本质上都是平台生态系统在其生命周期中的一次“活性行为”。这些行为共同构成了平台能力流动的“神经网络”，是平台价值输出的直接体现。

正如人类大脑通过神经元之间的连接实现信息传递与功能协作，那些高频使用的神经连接会被不断强化，而长期闲置或低频的连接则会被弱化，甚至“修剪”。这种自我优化机制，使大脑能够在有限资源下实现高效运行与持续进化。

在中心平台的建设中，我们也引入了类似的“运营反馈机制”。通过数据监测与行为分析，我们能够识别出哪些连接具有高价值、高频使用；哪些接口或服务已逐渐低频，甚至成为“死亡接口”。这些洞察可以为平台的持续演进提供依据。

## 外围应用建设原则

接下来，我们再来看一下外围应用的建设原则。

[![](https://static001.geekbang.org/resource/image/23/b6/23c0a347de0ca6644cyy60977fd7d4b6.jpg?wh=4000x2250)](https://static001.geekbang.org/resource/image/23/b6/23c0a347de0ca6644cyy60977fd7d4b6.jpg?wh=4000x2250)

原则一：场景化

外围应用的设计与开发应以真实业务场景为出发点，围绕用户实际的使用情境进行功能构建与流程优化。实现“场景驱动、按需定制”，从而提升应用的实用性与落地效率。

原则二：轻量化

在满足场景化需求的前提下，外围应用应尽量实现轻量化，强调低资源消耗、快速启动与高效运行，具备短响应时间和灵活的更新替换能力。例如，可通过微卡片、智能体、FaaS（Function as a Service）等轻型应用形式实现。

同时，部分功能无需依赖传统的前后端开发模式，即可通过平台化能力直接构建。例如，创建流程型智能体（Agent）、编写量化回测脚本、训练 AI 模型等，均可在低代码甚至无代码的环境下完成。

随着上层应用创新频率的不断提升，其形态也日趋场景化和碎片化。在此背景下，应用轻量化和门槛降低将有助于业务方快速试错与高效创新。

总体而言，外围应用应具备高度的敏捷性，能够快速响应变化，支持持续迭代与快速上线，灵活应对新需求与突发问题。就像我们前面比喻的那样，它应像一个“U 盘”，可以随时插拔于中心平台，实现即插即用、灵活扩展。

原则三：体验化

在用户交互和视觉设计方面，外围应用应紧跟大模型等技术趋势，关注用户习惯和感知体验。

在当前实践中，一个突出的问题是业务流程之间的割裂，导致用户体验缺乏连贯性。为解决这一问题，我们正从多个方面进行优化与重构。

一是在个人工作台中，支持用户可根据需求自由组合微卡片，自定义流程。二是利用事件驱动机制，通过对关键业务事件的梳理与建模，实现上下游流程的自动衔接，推动业务动作的自然流转。三是借助智能推荐技术，系统可根据用户的行为习惯与偏好，主动推荐相关的卡片或流程组合。

## 两者共同建设原则

[![](https://static001.geekbang.org/resource/image/fb/d7/fbdbb928127181030c1a6985dfb5bdd7.jpg?wh=7843x3798)](https://static001.geekbang.org/resource/image/fb/d7/fbdbb928127181030c1a6985dfb5bdd7.jpg?wh=7843x3798)

一、智能化

在前面的课程中，我们多次强调了面向智能体的设计理念，适用于从整体架构到中心平台、外围应用等不同层次与粒度的设计中，这里就不再重复了。

二、洞察化

中心平台与外围应用应按照“记忆框架”的要求，系统化地采集各类数据，并通过统一的数据中台实现跨应用的数据整合与分析。

然后，基于统一的数据资产，可以为业务、产品、开发、管理层等不同角色提供一致的分析视图。同时，也支持各个角色基于自身需求直接面向数据或接口进行个性化分析，提升数据的使用价值。

三、闭环化

在中心平台与外围应用中，会涉及多个层次和维度的闭环，这本质上是一种“端到端”的思维模式。

比如，感知 - 决策闭环：通过外围应用采集用户行为、业务操作、系统状态等多维数据，实时反馈至中心平台，驱动智能分析与决策；同时，中心平台输出的策略和控制指令又能快速下发至外围应用，形成“感知 - 分析 - 决策 - 执行”的完整闭环链条。

需要特别强调的是，由于外围应用直接面向用户，闭环的最终落脚点往往应该落在外围应用中，以实现价值的充分释放。

## 总结

在这节课中，我们围绕“业务 + 科技”共建的未来趋势，提出了一种全新的架构范式——“八爪鱼架构”。该架构由两个核心层次构成：中心平台层和外围应用层，并分别探讨了它们的建设原则。

[![](https://static001.geekbang.org/resource/image/f7/e2/f73d15cb80581ae92a253c75bbd6fbe2.jpg?wh=4325x4177)](https://static001.geekbang.org/resource/image/f7/e2/f73d15cb80581ae92a253c75bbd6fbe2.jpg?wh=4325x4177)

在未来的发展中，中心平台将成为科技建设与维护的核心。它不仅要承载企业共性能力的构建，更要将这些能力通过数字化手段，灵活地输出并投射到各种业务场景中，实现科技赋能业务的目标。

此外，外围应用是科技与业务深度融合的“前线阵地”。随着大模型时代的到来，业务对敏捷性、灵活性和创新能力的要求将不断提升。因此，未来外围应用将主要以轻量级的微卡片、智能体等形式存在。这类应用具有“随时创建、即刻使用”的特点，特别适合应对碎片化、动态变化的业务需求。

当然，对于“八爪鱼架构”这种双层架构模式，可能会有人提出疑问——随着企业系统生态的不断复杂化，这样的架构是否足够支撑？会不会导致系统越来越混乱？

对此，需要明确的是，“八爪鱼架构”是企业中最顶端的一层架构，它的目标不仅服务于科技团队，更重要的是让业务团队也能听得懂、看得见、参与得进来。因此，这种架构在设计上必须追求极致简洁，并具备强烈的“画面感”，让业务人员能够直观理解，从而实现真正的“共建”。

需要强调的是，在更细化的架构层面，科技可以继续进行分层设计与技术演进。但这些复杂性应由科技团队内部消化，而不是暴露给业务端。

最后，还有一个关键问题是：中心平台究竟应该如何建设？它与以往的平台或中台，又有什么本质区别？我们下节课还会深入探讨这一主题。
