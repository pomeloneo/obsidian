## 1.1 大模型的“情感表达能力”从何而来？

Science 2002 年 11 月发表题为《Emotion，cognition，and behavior》的论文，论文中指出，情感认知能力在人类社会中泛指感知、识别和理解他人情感的能力。但受限于对人机交互机理的认知不够清晰，在计算机视域下该能力收敛于“情感计算”。该概念由麻省理工学院媒体实验室 Picard 于 1997 年提出，旨在以计算为手段，研究和开发能够识别、解释和处理人类情感系统的一个跨学科领域，涉及计算机科学、心理学和认知科学。

情感计算的基础是情感量化模型，主要用于将无形的、抽象的人类情感转化为机器人可以理解、表达和处理的数据形式。ChatGPT 等基于 Transformer 结构的大语言模型语义理解能力的阶跃式提升对单模态情感认知产生了冲击。大模型本身并不具备情感，但它通过海量人类对话数据的学习，掌握了情感表达的语言模式。例如：

安慰语句常包含“我理解”“你并不孤单”“慢慢来”等词汇。

鼓励语句多使用积极动词和肯定结构，如“你可以的”“我相信你”。

共情表达倾向于使用感受类词汇，如“委屈”“疲惫”“无助”。

这些模式构成了 AI“拟人化表达”的基础。但问题在于：模型不知道什么时候该用哪种模式。这正是开发者需要介入的地方。

## 1.2 响应生成的三大挑战

在人文主义传统中，人拥有情感是一种自明的自我理解。人是情感的存在物，情感则是人对对象的感受和体验。发展到人机传播，人的感受和体验将与人工智能的理解、生成和回应情感的能力紧密相关。机器通过与用户的长期互动，逐步形成准确识别、理解用户情感的能力，生成带有情感色彩的文本、语音等多模态内容，人机之间进而产生深层次情感关系。

如图 12-1 所示，人类与自己创造出的无意识但又极为强大的非人类智能共存时，必须时刻审视向内的心灵成长和内向超越，自我修正情感的偏执、茧化、极化，甚至是疯狂。在人机交流的实践中，关心和提升人类大脑、身、心、灵的整合发展，找到心之所归。

在实际开发中，我们会遇到以下典型问题：

情感错配：用户表达悲伤，AI 却回复轻松调侃；

过度共情：AI 表现得比用户还激动，反而造成压迫感；

内容空洞：回复看似温暖，实则缺乏实质信息，如“抱抱你”反复出现。

这些问题的根源，往往不是模型能力不足，而是提示工程与上下文设计的缺失。我们需要一套系统方法，将“情感意图”转化为“可执行的生成策略”。

# 三、情感一致性校验：让 AI“说到做到”

即使我们精心设计了提示词，大模型仍可能“跑偏”。例如：

用户处于悲伤情绪，AI 却回复“哈哈，这有什么好难过的”。

连续对话中，AI 突然改变语气或角色设定。

为此，我们需要引入情感一致性校验机制。

## 3.1 基于小模型的情绪分类器

我们可以训练一个轻量级情绪分类器（如 BERT 微调版），对 AI 生成的回复进行二次检测，判断其情绪倾向是否与用户输入匹配。

ef check_emotion_consistency(self, ai_response: str, user_emotion: str) -> Tuple[bool, str]:

“““检查AI回复的情绪与用户情绪是否一致”“”

response_emotion, confidence = self.detect_emotion(ai_response)

forbidden_combinations = [

(“sad”, “happy”),

(“anxious”, “excited”),

(“high_risk_depression”, “happy”)

]

for forbidden in forbidden_combinations:

if (user_emotion in forbidden and response_emotion in forbidden):

return False, f”严重情感错配：用户{user_emotion}，AI回复{response_emotion}”

compatible_emotions = self.EMOTION_COMPATIBILITY.get(user_emotion, [“neutral”])

if response_emotion in compatible_emotions:

return True, “情感匹配”

return False, f”情感不匹配：用户{user_emotion}，AI回复{response_emotion}”

完整实现请参考：backend/utils/sentiment_classifier.py

若不一致，则触发降级策略，使用更安全的预设回复，或重新生成。

## 3.2 角色稳定性监控

通过关键词监控和语义相似度计算，确保 AI 不偏离预设角色。例如：

禁止使用“我是 GPT”“我是一个 AI”等暴露身份的表述；

避免突然切换语气（如从温柔变为严肃）。

def _post_process_response(self, response: str, user_emotion: str) -> str:

“““后处理生成的回复”“”

identity_replacements = {

“我是AI”: “我是心语”,

“我是一个AI”: “我是心语”,

“作为AI”: “作为陪伴者”

}

for old, new in identity_replacements.items():

if old in response:

response = response.replace(old, new)

return response

可在日志中记录每次回复的“角色偏离度”，用于后续优化。

## 4.4 进阶优化：让共情更“智能”

引入用户画像增强个性化，结合向量数据库中的长期记忆，提取用户偏好（如“喜欢温柔语气”“抗拒建议”），进一步微调回复风格。

多轮情感轨迹分析，不只看单次情绪，而是分析用户情绪变化趋势。例如：

持续低落 → 启动积极引导策略；

情绪逐步好转 → 加强正向反馈。

A/B 测试不同策略，部署多个版本的响应策略，通过用户停留时长、满意度评分等指标，持续优化最佳方案。

# 思考题

用户在多轮对话中呈现了如下情绪变化轨迹：

初始输入：“下周要参加职业资格考试，还有好多知识点没看完，越看越慌，晚上都睡不着”（情绪识别为 “焦虑”，强度 6.5/10）；

补充输入：“刚才做模拟题错了一半，感觉复习了这么久都白忙活了，特别挫败”（情绪转为 “沮丧”，强度 7.8/10）；

最终输入：“其实我已经考了两次都没通过，这次再不过，感觉自己真的很没用”（情绪升级为 “悲伤 + 轻度自我否定”，强度 8.2/10，无高风险倾向）。

请结合今天所学的“感知 — 决策 — 生成”响应链路、情感 - 策略映射、动态 Prompt 设计、混合生成架构及情感一致性校验等核心技术，回答以下问题：

针对用户每一轮的情绪状态，你会为“心语”机器人选择对应的响应策略（参考这一讲中的情感 - 策略映射表），并说明选择理由（需体现情绪强度对策略的影响）？

请为第三轮（悲伤 + 轻度自我否定）设计完整的动态 Prompt，明确其中的关键变量（情绪标签、回应策略、语气要求、对话历史摘要等）如何落地？
